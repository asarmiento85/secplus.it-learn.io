# 4.9 — Data Sources for Investigations

## What you need to know

Effective security investigations depend on knowing where to look for evidence and how to interpret what you find. Security+ tests your ability to identify the right data source for a given investigation scenario, understand what specific log types and data sources reveal, and recognize key Windows Event IDs. You should also understand the difference between packet captures (full content) and NetFlow (metadata only), and know why correlating multiple data sources is essential to building a complete picture.

---

## Log Types

### Firewall Logs
- Record: **allow and deny decisions** for network traffic
- Include: source IP, destination IP, source port, destination port, protocol, action (allow/deny), timestamp
- Use in investigations: identify attacker source IPs, confirm traffic patterns, verify whether blocked traffic reached destination
- Can confirm **data exfiltration** (large outbound transfers to unknown IPs)
- Limitations: do not capture packet contents; may be voluminous (millions of entries per day)

### Application Logs
- Record: **application-level events** — errors, transactions, user actions, authentication events
- Examples: web server access logs (Apache, IIS), database query logs, application error logs
- Use in investigations: identify failed login attempts, track user transactions, find application-level attacks (SQLi, XSS)
- Web access logs include: client IP, requested URL, response code, user-agent, bytes transferred

### Endpoint / Host Logs

#### Windows Event Log — Critical Event IDs

| Event ID | Description |
|---|---|
| **4624** | Successful logon |
| **4625** | Failed logon |
| **4634 / 4647** | Logoff |
| **4648** | Logon using explicit credentials (Run As, Pass-the-Hash indicator) |
| **4672** | Special privileges assigned (admin logon) |
| **4688** | New process created — track execution of processes |
| **4698** | Scheduled task created (common malware persistence) |
| **4720** | User account created |
| **4722** | User account enabled |
| **4723 / 4724** | Password change attempt |
| **4726** | User account deleted |
| **4732 / 4733** | User added to / removed from a security group |
| **4776** | Domain controller attempted to validate credentials |
| **7045** | New service installed (common malware persistence) |

- Windows logs stored in: **Security** (authentication/authorization), **System** (OS events), **Application** (app events)
- **Sysmon** (System Monitor) — free Microsoft tool that logs rich endpoint telemetry (process creation, network connections, file changes)

### OS Logs (Linux/Unix)
- **Syslog** — standard Linux logging facility; different severity levels (emergency through debug)
- Common log files:
  - `/var/log/auth.log` or `/var/log/secure` — authentication events (sudo, SSH logins)
  - `/var/log/syslog` or `/var/log/messages` — general system events
  - `/var/log/apache2/access.log` — web server access
  - `/var/log/kern.log` — kernel events
- **Journald** — modern systemd-based logging; query with `journalctl`

### IPS/IDS Logs
- Record: **signatures triggered**, source/destination, severity, timestamp
- Use in investigations: identify attack patterns, lateral movement, exploit attempts
- Can show: which vulnerabilities attackers tried to exploit, attack timeline

### Network Logs

| Log Type | What It Captures | Use Case |
|---|---|---|
| **NetFlow** | Traffic metadata (who talked to whom, when, how much data) | Anomaly detection, connection mapping, exfiltration detection |
| **DNS query logs** | Domain name lookups — who queried what | Detect C2 beaconing, malicious domain lookups, tunneling |
| **DHCP logs** | IP address assignments — MAC to IP mapping | Attribute network activity to specific device |
| **Proxy logs** | Web requests through proxy — full URL, user, response | Detailed web browsing history, URL analysis |
| **VPN logs** | Who connected, from where, when, duration | Detect unauthorized access, verify alibi |
| **Flow data** | Similar to NetFlow; various implementations | Traffic analysis at scale |

## Metadata

- **Metadata** = data about data; often overlooked but valuable in investigations

| Metadata Type | Examples |
|---|---|
| **File metadata** | Creation time, modification time, last access time, author, file size |
| **Email metadata** | Sender, recipients, server path (Received headers), timestamps, IP addresses |
| **Photo/image metadata (EXIF)** | GPS coordinates, camera make/model, timestamp, settings |
| **Document metadata** | Author, organization, template, revision history, tracked changes |
| **Network metadata** | Source IP, dest IP, port, protocol, bytes, timestamps (NetFlow) |

- **EXIF data** in photos can reveal GPS location — privacy risk; used in forensics
- Email **Received headers** show the path the email took across mail servers — useful for tracing spam/phishing origin
- Timestamps (MTIME/CTIME/ATIME) on files are key forensic artifacts — but can be manipulated (timestomping)

## Data Sources Comparison

| Source | Content | Level of Detail | Use Case |
|---|---|---|---|
| **PCAP (packet capture)** | Full packet contents including payload | Highest | Detailed forensic analysis, malware traffic, credentials in plaintext |
| **NetFlow** | Metadata only: IPs, ports, protocol, bytes, duration | Medium | Traffic analysis, anomaly detection, exfiltration volume |
| **Logs** | Structured events from systems/apps | Variable | Authentication, access, error tracking |
| **SIEM dashboards** | Aggregated, normalized view of log data | Summary | Real-time monitoring, trend analysis |
| **Vulnerability scan results** | Known vulnerabilities on scanned systems | Technical | Risk prioritization, remediation tracking |
| **Automated reports** | Scheduled compliance and trend reports | Summary | Management reporting, compliance evidence |

## Packet Captures (PCAP)

- **Wireshark** — graphical packet analyzer; most widely used
- **tcpdump** — command-line packet capture: `tcpdump -i eth0 -w capture.pcap`
- PCAP files capture: every packet's full content including payload
- Use for: malware C2 traffic analysis, credential capture in plaintext protocols, reconstructing file transfers
- Limitations: massive file sizes; requires storage planning; raises privacy concerns
- **Write blocker + forensic copy** required for legal admissibility of captured evidence

## SIEM Dashboards and Automated Reports

### SIEM Dashboards
- **Real-time visibility** into security events and alerts
- Common dashboard views: top alerting systems, authentication failures by user, geographic login map, threat severity over time
- Custom dashboards: tailored to team role (SOC analyst vs. CISO vs. compliance)

### Automated Reports
- **Scheduled compliance reports** — generated automatically at defined intervals
- Examples: weekly failed login summary, monthly patch compliance status, quarterly access review report
- Provide management with consistent metrics without manual effort
- Feed into GRC (Governance, Risk, and Compliance) programs

## Correlation: Putting It All Together

Single data sources rarely tell the complete story. Effective investigations correlate multiple sources.

**Example Investigation — Ransomware Incident:**

| Data Source | What It Shows |
|---|---|
| SIEM alert | Unusual volume of file modification events |
| Windows Event Log (4688) | New process `powershell.exe` spawned by `word.exe` |
| Firewall logs | Outbound connection to known malicious IP after process creation |
| DNS logs | Lookup for malicious domain 5 minutes before encryption started |
| NetFlow | Large outbound transfer (data exfiltration) before encryption |
| Endpoint DLP | Alert on bulk file access matching ransomware behavior |

- Combining all sources creates a **timeline** and confirms the attack chain
- **SIEM correlation rules** automate multi-source correlation in real time

## Exam Tips

- **NetFlow captures metadata only** — not packet contents. Use **PCAP/Wireshark** for full content.
- Know the key Windows Event IDs: **4624** (logon), **4625** (failed logon), **4688** (new process), **4720** (account created).
- **4648** — explicit credentials — often indicates Pass-the-Hash or lateral movement.
- **DNS logs** are critical for detecting C2 beaconing and DNS tunneling.
- **DHCP logs** map IP addresses to physical devices — essential for attribution.
- **EXIF metadata** in photos can contain **GPS coordinates** — a privacy and forensic data point.
- **Syslog** is the standard Linux logging protocol; know common `/var/log/` paths.
- **Correlation across multiple sources** is always superior to relying on a single source.
- **Automated reports** = scheduled, recurring compliance and trend reports — not the same as real-time dashboards.
- **Sysmon** provides much richer Windows endpoint telemetry than standard Windows event logging.
- Investigating with only firewall logs is insufficient — they show connections, not what was executed or accessed.
- **Timestamps** in logs must be synchronized via **NTP** to correlate events across systems accurately.
