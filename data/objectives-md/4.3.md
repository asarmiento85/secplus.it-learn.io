# 4.3 — Vulnerability Management

## What you need to know

Vulnerability management is the continuous process of identifying, analyzing, responding to, and validating security weaknesses across systems and applications. The exam tests every phase of the cycle — identification methods, analysis with CVSS scoring, response strategies, and validation. Know CVSS metrics cold, understand how to prioritize vulnerabilities, and be prepared for scenario-based questions asking what a SOC analyst should do when a vulnerability is discovered.

---

## Vulnerability Identification Methods

| Method | Description | Notes |
|---|---|---|
| **Vulnerability scans** | Automated tools probe systems for known weaknesses | Nessus, Qualys, OpenVAS; authenticated scans are more thorough |
| **Application security testing** | SAST/DAST/penetration testing of applications | Finds app-layer flaws |
| **Threat feeds** | External intelligence about new vulnerabilities | CISA KEV, MITRE CVE, vendor advisories |
| **Penetration testing** | Human tester simulates attacker to discover exploitable vulnerabilities | More thorough than automated scans |
| **Responsible disclosure / Bug bounty** | External researchers report vulnerabilities to the organization | Coordinated disclosure; bug bounties incentivize researchers |
| **Audits** | Formal review of configurations and controls against standards | Internal or third-party; checks compliance posture |

### Vulnerability Scan Types

| Type | Description |
|---|---|
| **Authenticated (credentialed)** | Scanner uses valid credentials; sees more detail, fewer false positives |
| **Unauthenticated (non-credentialed)** | Scanner uses no credentials; simulates external attacker view |
| **Internal** | Run from inside the network; discovers internal vulnerabilities |
| **External** | Run from outside; discovers perimeter-facing vulnerabilities |
| **Passive** | Listens to traffic; no active probing; stealthy |
| **Active** | Actively probes/sends packets to targets; more complete results |

## Vulnerability Analysis

### Confirmation — Avoiding False Positives

- **False positive** — scanner reports a vulnerability that doesn't actually exist
  - Cause: banner grabbing, OS fingerprinting errors, plugin bugs
  - Action: manually verify before reporting or remediating
- **False negative** — scanner misses a real vulnerability
  - Cause: authenticated scans may be required; scanner plugin not updated
  - Risk: organization believes it's secure when it isn't
- **True positive** — confirmed real vulnerability; requires remediation
- Manual confirmation involves: examining the system, checking patch level, attempting a controlled exploit (pen test)

### CVSS — Common Vulnerability Scoring System

CVSS provides a standardized 0–10 score representing vulnerability severity.

| CVSS Score Range | Severity |
|---|---|
| 0.0 | None |
| 0.1 – 3.9 | Low |
| 4.0 – 6.9 | Medium |
| 7.0 – 8.9 | High |
| 9.0 – 10.0 | Critical |

#### CVSS Base Metrics

| Metric | Abbreviation | Options |
|---|---|---|
| **Attack Vector** | AV | Network (N), Adjacent (A), Local (L), Physical (P) |
| **Attack Complexity** | AC | Low (L), High (H) |
| **Privileges Required** | PR | None (N), Low (L), High (H) |
| **User Interaction** | UI | None (N), Required (R) |
| **Scope** | S | Unchanged (U), Changed (C) |
| **Confidentiality Impact** | C | None (N), Low (L), High (H) |
| **Integrity Impact** | I | None (N), Low (L), High (H) |
| **Availability Impact** | A | None (N), Low (L), High (H) |

- **Higher score** = Network AV, Low AC, No PR, No UI, Changed Scope, High C/I/A
- **Lower score** = Physical AV, High AC, High PR, Required UI, Unchanged Scope, Low C/I/A
- CVSS also has **Temporal** (exploit maturity, remediation availability) and **Environmental** (org-specific adjustments) metric groups

### CVE Identifiers
- **CVE** (Common Vulnerabilities and Exposures) — unique identifier for a known vulnerability
- Format: **CVE-YYYY-NNNNN** (e.g., CVE-2021-44228 = Log4Shell)
- Maintained by MITRE; published in National Vulnerability Database (NVD)
- NVD enriches CVE with CVSS scores, affected product info, and patch links

### Prioritization Factors

| Factor | Description |
|---|---|
| **CVSS score** | Higher score = higher priority |
| **Exploitability** | Is a public exploit available? CISA KEV list? |
| **Asset criticality** | Is the affected system business-critical? |
| **Exposure** | Is the system internet-facing? Internal only? |
| **Environmental variables** | Compensating controls in place? Network segmentation? |
| **Risk tolerance** | Organization's appetite for accepting vs. remediating risk |
| **Exposure factor** | What percentage of the asset's value is at risk? |

## Vulnerability Response Strategies

| Strategy | Description | When to Use |
|---|---|---|
| **Patching** | Apply vendor-supplied fix | Preferred; address root cause |
| **Insurance** | Cyber insurance to transfer financial risk | Cannot eliminate risk; need financial protection |
| **Segmentation** | Isolate vulnerable system from rest of network | Cannot patch (legacy/ICS); limits blast radius |
| **Compensating controls** | Alternative controls that reduce risk | When patching not feasible |
| **Exceptions / Exemptions** | Formally accept risk with documented justification | When all other options are not viable |

- **Patching** is always the preferred response — it eliminates the vulnerability
- **Exceptions** require: risk owner sign-off, business justification, compensating controls, expiration date, and regular review
- **Virtual patching** — IPS/WAF rules that block exploitation of a vulnerability while a patch is developed

---

## Validation After Remediation

- **Rescan** — run the vulnerability scanner again to confirm the patch was applied and the finding is resolved
- **Audit verification** — third-party or internal auditor confirms remediation
- **Penetration test** — verify the vulnerability is no longer exploitable (not just patched)
- **Change management verification** — confirm the patch was logged in the CMDB

---

## Reporting to Stakeholders

- **Technical report** — detailed findings for IT/security team (CVE IDs, CVSS scores, affected systems)
- **Executive summary** — risk-based language for management (business impact, remediation progress, trend)
- **Metrics**: Mean Time to Remediate (MTTR), % of criticals patched within SLA, vulnerability trend over time
- Reports should show **trend data** — are we improving or getting worse over time?

## Exam Tips

- **CVSS 9.0-10.0 = Critical** — highest priority for remediation.
- A **false positive** means the scanner flagged something that isn't actually vulnerable — always verify before acting.
- A **false negative** is more dangerous than a false positive — you don't know you're vulnerable.
- **CVE** = identifier for a specific vulnerability; **CVSS** = the scoring system; **NVD** = the database.
- **Patching is always the preferred response** — every other option (segmentation, compensating controls, exceptions) is a workaround.
- **Exceptions require documentation and justification** — accepting risk is not the same as ignoring it.
- Know the CVSS **Attack Vector** values: Network > Adjacent > Local > Physical (Network = highest risk, most remote).
- **CISA KEV** (Known Exploited Vulnerabilities) catalog = real-world exploited vulnerabilities requiring priority remediation.
- An **authenticated scan** will find more vulnerabilities and produce fewer false positives than an unauthenticated scan.
- **Scope = Changed** in CVSS means exploiting the vulnerable component can affect other systems — critical factor in scoring.
- After remediation: **always rescan** to confirm the vulnerability is truly resolved.
