# 2.3 — Types of Vulnerabilities

## What you need to know

A **vulnerability** is a weakness in a system, application, or process that can be exploited by a threat actor. The Security+ exam covers a wide range of vulnerability categories — from application code flaws to hardware weaknesses to human-process gaps. For each type, you need to understand what it is, how it is exploited, and the key mitigation. Knowing the exact terminology (TOCTOU, VM escape, zero-day, etc.) is critical for exam success.

---

## Application Vulnerabilities

### Memory Injection

- An attacker injects malicious code directly into the memory space of a running process
- The injected code executes with the permissions of the target process
- Common in process hollowing and DLL injection attacks
- Used by malware to hide inside legitimate processes (e.g., injecting into svchost.exe)

### Buffer Overflow

| Aspect | Detail |
|---|---|
| **What it is** | Writing data beyond the allocated memory buffer, overwriting adjacent memory |
| **Impact** | Can overwrite return addresses → arbitrary code execution; crash the application |
| **Classic targets** | C/C++ programs that don't perform bounds checking |
| **Defenses** | Address Space Layout Randomization (ASLR), Data Execution Prevention (DEP), stack canaries, safe coding practices |

- Buffer overflow is one of the **oldest and most well-known** vulnerability classes
- Stack-based: overwrites the return address on the call stack
- Heap-based: overwrites heap memory structures

### Race Conditions — TOCTOU

| Aspect | Detail |
|---|---|
| **Full name** | Time-of-Check to Time-of-Use |
| **What it is** | The state of a resource changes between when it is checked and when it is used |
| **Example** | Check file permissions → attacker swaps file → use (now wrong) file |
| **Impact** | Privilege escalation, unauthorized access, data corruption |
| **Defense** | Atomic operations, file locking, mutex controls |

- TOCTOU is a **timing attack** — exploits the window between security check and action
- Common in OS kernel code and privileged application logic

### Malicious Update

- Compromised software update mechanism delivers malware instead of (or in addition to) a legitimate update
- Exploits the user's trust in the update process
- Examples: SolarWinds (update server compromised), NotPetya (fake M.E.Doc update)
- Defense: code signing, verify update signatures before installation, SBOM

## OS-Based Vulnerabilities

- **Unpatched OS**: missing security patches leave known CVEs exploitable
- **Insecure default configurations**: OS ships with unnecessary services enabled, weak default settings
- **Kernel vulnerabilities**: exploitation of kernel bugs can lead to full system compromise (ring 0 access)
- **Privilege escalation**: vulnerabilities that allow a low-privilege user to gain admin/root access
- Defense: regular patching, OS hardening (CIS benchmarks, STIGs), least privilege

## Web-Based Vulnerabilities

### SQL Injection (SQLi)

| Aspect | Detail |
|---|---|
| **What it is** | Attacker inserts malicious SQL code into input fields interpreted by a database |
| **Impact** | Authentication bypass, data theft, data modification, database destruction |
| **Types** | Union-based, blind (boolean/time-based), error-based |
| **Prevention** | Parameterized queries (prepared statements), stored procedures, input validation, WAF |

- Classic example: entering `' OR '1'='1` in a login field to bypass authentication
- The **single quote (')** is the telltale sign of a SQLi attempt in logs
- Blind SQLi infers data without direct output — slower but often works when error output is suppressed

### Cross-Site Scripting (XSS)

| Type | Description |
|---|---|
| **Reflected XSS** | Malicious script in URL; executes when victim clicks link; not stored on server |
| **Stored XSS** | Malicious script saved to database; executes for every user who loads the affected page |
| **DOM-based XSS** | Payload manipulates the DOM in the browser; never sent to the server |

- Attacker injects JavaScript that runs in victim's browser — can steal cookies, redirect, capture keystrokes
- Prevention: output encoding, Content Security Policy (CSP), input validation
- Stored XSS is the most dangerous type (affects all users who view the page)

### Cross-Site Request Forgery (CSRF)

- Tricks an **already-authenticated** user into making an unwanted request to a trusted site
- Example: victim is logged into their bank; attacker's page silently submits a transfer request using the victim's session
- Prevention: **CSRF tokens** (anti-forgery tokens), SameSite cookie attribute, requiring re-authentication for sensitive actions
- CSRF is about **abusing trust the server has in the user's browser**, while XSS abuses trust the user has in the site

### Directory Traversal (Path Traversal)

- Attacker uses `../` sequences to navigate outside the intended web root
- Example: `https://example.com/files?name=../../../../etc/passwd`
- Can expose configuration files, credentials, and sensitive OS files
- Prevention: input validation, canonicalize paths before use, web server configuration

## Hardware Vulnerabilities

| Type | Description |
|---|---|
| **Firmware vulnerabilities** | Bugs in embedded code (BIOS/UEFI, router firmware, IoT firmware) |
| **End-of-life hardware** | Vendor no longer provides firmware updates; vulnerabilities are permanent |
| **Legacy hardware** | Outdated equipment that cannot support modern security controls |
| **Side-channel attacks** | Exploiting physical characteristics (power consumption, EM emissions, timing) rather than software flaws — e.g., Spectre, Meltdown |

- Firmware attacks can persist through OS reinstallation (difficult to remediate)
- Legacy hardware often cannot run current OS versions — creates compatibility-driven security gaps

---

## Virtualization Vulnerabilities

| Vulnerability | Description |
|---|---|
| **VM escape** | Attacker breaks out of a virtual machine and gains access to the hypervisor or host OS |
| **Resource reuse** | Data left in memory or storage after a VM is deleted; next tenant may access previous tenant's data |
| **Side-channel attacks** | VMs on the same physical host can potentially observe each other's CPU cache patterns (Spectre) |
| **Hypervisor vulnerabilities** | Bugs in the hypervisor (Type 1: bare-metal; Type 2: hosted) can affect all guest VMs |

- VM escape is rare but considered the most severe virtualization vulnerability
- Resource reuse is a concern in multi-tenant cloud environments
- Defense: keep hypervisor software patched, use dedicated hardware for high-sensitivity workloads

---

## Cloud-Specific Vulnerabilities

| Vulnerability | Description |
|---|---|
| **Misconfigured storage** | S3 buckets or Blob storage set to public; exposes sensitive data |
| **Over-permissive IAM** | Cloud identity roles with more permissions than needed (violates least privilege) |
| **Shared responsibility gaps** | Misunderstanding of what the cloud provider secures vs. what the customer must secure |
| **Insecure APIs** | Cloud management APIs exposed without proper authentication |

- Most cloud breaches are due to **misconfiguration**, not sophisticated exploits
- Defense: cloud security posture management (CSPM), regular IAM reviews, enable object-level logging

---

## Supply Chain Vulnerabilities

- **Compromised components**: malicious hardware or software inserted during manufacturing
- **Software dependencies**: vulnerable open-source libraries included in applications
  - Log4Shell (CVE-2021-44228): critical vulnerability in the Log4j logging library; affected millions of Java apps
  - XZ Utils backdoor (2024): malicious code inserted into a widely-used compression library
- **Third-party software**: vendor's software may have vulnerabilities that affect customer environments
- Defense: software bill of materials (SBOM), dependency scanning, vendor vetting, code signing

---

## Cryptographic Vulnerabilities

| Weakness | Description |
|---|---|
| **Weak algorithms** | MD5 (hashing), DES/3DES (encryption) — considered broken or insufficient |
| **Short key lengths** | 512-bit RSA, 56-bit DES — too short to resist modern brute-force |
| **Improper implementation** | Correct algorithm, but poor implementation (e.g., reusing IVs, hardcoded keys) |
| **Deprecated protocols** | SSL 3.0, TLS 1.0/1.1, SSHv1 — vulnerable to known attacks |
| **Certificate issues** | Expired, self-signed, or mis-issued certificates |

- Use AES-256, SHA-256/SHA-3, RSA 2048+ or ECC, TLS 1.2/1.3
- Even strong algorithms fail if improperly implemented

---

## Misconfiguration Vulnerabilities

- **Default passwords** left unchanged
- **Unnecessary services** running (increases attack surface)
- **Open firewall rules** allowing excessive traffic
- **Debug mode** left enabled in production applications (exposes stack traces, source code)
- **Overly permissive ACLs** on files and directories
- **Missing security headers** in web servers (CSP, X-Frame-Options, HSTS)
- Defense: CIS benchmarks, STIG compliance, configuration management tools (Ansible, Puppet)

---

## Mobile Vulnerabilities

| Vulnerability | Description |
|---|---|
| **Outdated OS** | Mobile OS not updated; known CVEs exploitable |
| **Sideloaded apps** | Apps installed outside the official app store; no vetting for malware |
| **Jailbreaking / Rooting** | Removes OS security restrictions; may disable security features |
| **SMS-based attacks** | SIM swapping, smishing, SS7 protocol exploitation |
| **Insecure Wi-Fi** | Mobile devices auto-connecting to rogue access points |

- MDM (Mobile Device Management) enforces patching and prevents sideloading
- Jailbroken/rooted devices cannot be trusted by MDM policies

---

## Zero-Day Vulnerabilities

| Attribute | Detail |
|---|---|
| **Definition** | A vulnerability that is **unknown to the vendor** — no patch exists yet |
| **Value** | Extremely high — nation-states pay millions for exclusive zero-days |
| **Window of exposure** | From discovery until patch release (and deployment) |
| **Defense** | Defense-in-depth, behavior-based detection, network segmentation, rapid patching when patch is released |

- "Zero-day" refers to the fact that developers have had **zero days** to fix the flaw
- Once a vendor releases a patch, the vulnerability is no longer a zero-day — but unpatched systems remain vulnerable
- Zero-day exploits are a hallmark of nation-state and sophisticated organized crime actors

## Exam Tips

- **Buffer overflow** → can lead to arbitrary code execution by overwriting the return address — classic exam topic
- **TOCTOU** = race condition; remember "Time-of-Check to Time-of-Use" in full for scenario questions
- **SQLi prevention** = parameterized queries / prepared statements — not just input validation alone
- **XSS** injects into the browser; **CSRF** abuses the server's trust in the browser — know the difference
- **Stored XSS** is more severe than reflected because it affects all users who load the page
- **VM escape** is the most severe virtualization vulnerability — attacker breaks out of guest to host
- **Zero-day** = no patch exists yet; defense-in-depth is the key mitigation strategy
- **Log4Shell** is the go-to supply chain/dependency vulnerability example for SY0-701
- **Firmware vulnerabilities** can survive OS reinstallation — hardware replacement may be required
- **Misconfiguration** is responsible for the majority of cloud breaches — not sophisticated attacks
