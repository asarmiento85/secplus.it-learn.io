# 5.5 — Audits and Assessments

## What you need to know

Audits and assessments provide objective evidence that security controls are operating effectively. The exam tests your ability to distinguish between internal and external audits, understand SOC report types (especially SOC 2 Type I vs. Type II), and identify the appropriate penetration testing methodology for a given scenario. You need to know the three knowledge environments (white box, black box, grey box), the difference between red/blue/purple teams, and the distinction between active and passive reconnaissance. Penetration testing scenarios are common exam questions — focus on matching the testing type to the scenario.

---

## Attestation

- **Attestation** is a formal declaration by an authorized party that controls are in place and operating effectively
- An officer (CISO, CFO, CEO) signs a statement confirming compliance or control effectiveness
- Used in SOC reports, SOX filings, regulatory submissions, and vendor compliance confirmations
- Attestation creates legal accountability — signing falsely can result in criminal liability

---

## Internal Audits

Internal audits are performed by the organization's own audit staff or an internal security team.

| Type | Description |
|---|---|
| **Compliance Audit** | Verifies that the organization is following applicable regulations and internal policies |
| **Operational Audit** | Evaluates the effectiveness and efficiency of security operations |
| **Financial Audit** | Reviews financial controls and the accuracy of financial reporting |
| **Self-Assessment** | Department or team evaluates their own compliance against standards |
| **Audit Committee** | Board-level oversight body that reviews audit results and holds management accountable |

- Internal audits provide ongoing assurance but may lack independence
- **Audit committee** provides board-level oversight — typically independent directors
- **Self-assessments** are faster but less rigorous; good for gap identification before formal audits

---

## External Audits

External audits are performed by independent third parties — regulators, certification bodies, or independent auditors.

| Type | Description | Examples |
|---|---|---|
| **Regulatory Examination** | Government-mandated audit by a regulatory agency | OCC bank examinations, HHS HIPAA audits, SEC reviews |
| **Certification Audit** | Audit to achieve or maintain a certification | ISO 27001 certification, PCI-DSS QSA assessment |
| **Third-Party Audit** | Independent auditors evaluate controls and issue a report | SOC 1, SOC 2, SOC 3 reports |
| **Assessment** | Evaluation of security posture without formal certification | NIST CSF assessment, CIS Controls assessment |

---

## SOC Reports

SOC (System and Organization Controls) reports are issued by independent auditors and are critical for vendor assessments.

| Report | Purpose | Who Uses It |
|---|---|---|
| **SOC 1** | Controls relevant to financial reporting | Financial auditors, clients whose financials depend on the vendor |
| **SOC 2** | Security, Availability, Processing Integrity, Confidentiality, Privacy (Trust Service Criteria) | Security teams, procurement, enterprise clients |
| **SOC 3** | Publicly distributable summary of SOC 2 findings | Marketing, general public |

### SOC 2 Type I vs. Type II

| Type | Description | Strength |
|---|---|---|
| **SOC 2 Type I** | Point-in-time snapshot — confirms controls are **designed** appropriately at a single date | Lower assurance — controls may not operate consistently |
| **SOC 2 Type II** | Covers a period (typically 6–12 months) — confirms controls **operated effectively** throughout the period | Higher assurance — proves sustained control operation |

- **Type II is stronger** and preferred by security-conscious clients
- Achieving SOC 2 Type II requires time — organizations must operate controls consistently before the audit period
- **Trust Service Criteria (TSC)**: Security (required), Availability, Processing Integrity, Confidentiality, Privacy (optional add-ons)

## Penetration Testing

Penetration testing simulates real attacks to identify exploitable vulnerabilities before attackers do.

### Team Roles

| Team | Role | Description |
|---|---|---|
| **Red Team** | Offensive | Simulates attacker; attempts to compromise systems, bypass defenses, gain access |
| **Blue Team** | Defensive | Detects, responds to, and remediates attacks; operates security controls |
| **Purple Team** | Integrated | Red and blue teams work together; red shares findings in real-time to improve blue's detection capability |
| **White Team** | Oversight | Manages the exercise; sets rules of engagement; tracks findings; referee role |

### Testing Knowledge Levels

| Environment | Also Known As | Description | Best For |
|---|---|---|---|
| **Known Environment** | White Box | Tester has full knowledge — network diagrams, source code, credentials | Most thorough testing; finds deepest vulnerabilities; not realistic of external attacker |
| **Unknown Environment** | Black Box | Tester has no prior knowledge of target | Most realistic simulation of an external attacker; may miss some issues due to limited time |
| **Partially Known Environment** | Grey Box | Tester has some information (user account, limited network info) | Balance of realism and depth; common for internal assessments |

### Penetration Testing Types

| Type | Description | Techniques |
|---|---|---|
| **Network** | Test network infrastructure for vulnerabilities | Port scanning, service enumeration, exploitation of network services |
| **Web Application** | Test web apps for OWASP Top 10 and other flaws | SQL injection, XSS, CSRF, broken authentication testing |
| **Physical** | Attempt to gain unauthorized physical access | Tailgating, lock picking, badge cloning, dumpster diving |
| **Social Engineering** | Test human defenses through manipulation | Phishing calls, pretexting, impersonation |
| **Wireless** | Test wireless network security | WPA2 cracking, rogue AP detection, evil twin testing |
| **Cloud** | Test cloud environment configuration | IAM misconfigurations, storage bucket exposure, metadata endpoint testing |

---

## Reconnaissance

Reconnaissance is the information-gathering phase before active exploitation.

| Type | Description | Examples |
|---|---|---|
| **Passive Reconnaissance** | Gathering information without directly interacting with the target | OSINT (Google, LinkedIn, Shodan), DNS lookups, WHOIS, reviewing public code repos |
| **Active Reconnaissance** | Directly interacting with the target to gather information | Port scanning (Nmap), service enumeration, banner grabbing, vulnerability scanning |

- **Passive recon** cannot be detected by the target — no traffic reaches their systems
- **Active recon** generates network traffic and can be detected by IDS/IPS
- **OSINT** (Open Source Intelligence) is always passive — publicly available data sources only

---

## Pivot and Lateral Movement

| Concept | Description |
|---|---|
| **Pivot** | Using a compromised system as a launching point to attack other systems in the network |
| **Lateral Movement** | Moving through the network from one compromised host to others to expand access |
| **Privilege Escalation** | Gaining higher-level permissions on a compromised system (local admin → domain admin) |
| **Persistence** | Maintaining access to a compromised system after initial exploitation (backdoor, scheduled task) |

---

## Exam Tips

- **SOC 2 Type II is stronger than Type I** — Type II proves controls operated over a period; Type I is just a snapshot
- **White box = known environment** (most thorough); **Black box = unknown environment** (most realistic); **Grey box = partial knowledge**
- **Red team = offensive**; **Blue team = defensive**; **Purple team = both working together**
- **Passive reconnaissance** cannot be detected — OSINT, WHOIS, DNS lookups leave no trace on the target
- **Active reconnaissance** interacts with the target — port scans and vulnerability scans can be detected
- **Physical penetration testing** includes tailgating, lock picking, and social engineering at reception — tests physical controls
- **Attestation** is a formal signed declaration — used in SOC reports, SOX, and regulatory compliance submissions
- **SOC 3** is the publicly distributable summary of SOC 2 — no sensitive details; can be posted on a website
- **Pivot** = using one compromised host to attack others inside the network — common in advanced persistent threats
- **Self-assessments** are internal, faster, but less rigorous — useful for identifying gaps before a formal audit
