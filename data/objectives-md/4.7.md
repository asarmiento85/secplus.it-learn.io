# 4.7 — Automation and Orchestration in Security

## What you need to know

Security automation and orchestration enable security teams to respond faster, more consistently, and at scale. The exam tests your understanding of SOAR platforms, specific use cases for automation, the benefits security automation delivers, and the real-world considerations (risks and costs) of implementing automation. Know SOAR cold, be able to explain why automation is used, and understand both the benefits and the limitations.

---

## Core Concepts

| Term | Definition |
|---|---|
| **Automation** | Using technology to execute tasks without human intervention |
| **Orchestration** | Coordinating multiple automated tasks and tools to complete a workflow |
| **SOAR** | Security Orchestration, Automation, and Response — platform combining all three |
| **Playbook** | Documented workflow (manual) for responding to a security scenario |
| **Runbook** | Automated version of a playbook — executed by the SOAR platform |
| **IaC** | Infrastructure as Code — define infrastructure in code (Terraform, CloudFormation) |
| **CI/CD** | Continuous Integration / Continuous Delivery — pipeline for automated software delivery |

---

## SOAR (Security Orchestration, Automation, and Response)

SOAR platforms automate repetitive Security Operations Center (SOC) tasks, connecting security tools and automating response workflows.

### SOAR Capabilities

| Capability | Description |
|---|---|
| **Orchestration** | Integrates multiple security tools (SIEM, ticketing, threat intel, EDR) into a unified workflow |
| **Automation** | Executes actions without human intervention (block IP, isolate host, open ticket) |
| **Response** | Executes automated containment and remediation actions |
| **Case management** | Tracks incidents from detection through closure |
| **Threat intelligence** | Enriches alerts with threat feeds (VirusTotal lookups, IP reputation checks) |

### Example SOAR Workflow
1. SIEM fires alert: suspicious outbound connection detected
2. SOAR automatically queries threat intel (is this IP malicious?)
3. If confirmed malicious: SOAR blocks IP on firewall, isolates endpoint via EDR, opens ServiceNow ticket, notifies analyst via Slack
4. Analyst reviews, approves, and closes ticket
- What took 30 minutes of manual work → done in seconds automatically

### SOAR Platforms
- **Splunk SOAR** (formerly Phantom)
- **Microsoft Sentinel** (has built-in SOAR capabilities)
- **Palo Alto XSOAR** (formerly Demisto)
- **IBM Security SOAR** (formerly Resilient)

## Use Cases for Security Automation

### User and Resource Provisioning
- **Automated onboarding** — new hire ticket triggers account creation, email setup, device enrollment, app access
- **De-provisioning** — termination ticket triggers immediate account disabling, access revocation
- Reduces human error and ensures consistent application of policy

### Guard Rails
- **Automated policy enforcement** — if a cloud resource is misconfigured (e.g., S3 bucket set to public), automation immediately remediates
- Prevents security misconfigurations before they can be exploited
- Example: Azure Policy auto-remediates non-compliant resources

### Security Group Management
- Automatically add/remove users from AD groups based on role changes in HR system
- Ensures access reflects current job function in real time

### Ticket Creation and Escalation
- SIEM alert automatically creates an incident ticket with relevant context
- Escalation workflows: if ticket is not resolved within SLA, automatically escalate to senior analyst or management
- Reduces mean time to respond (MTTR)

### Enabling/Disabling Services
- Automatic service disabling: if a user account is flagged for anomalous behavior, disable service automatically pending investigation
- Automated firewall rules: block/unblock based on threat intelligence

### CI/CD Pipeline Security
- **Shift-left security** — integrate security scanning (SAST, SCA, secret scanning) into the pipeline
- Code fails build if vulnerabilities above threshold are found
- **DevSecOps** — security integrated throughout development, not bolted on at the end
- Automated container image scanning before deployment

### Infrastructure as Code (IaC)
- Define all infrastructure in code (Terraform, Ansible, CloudFormation)
- Benefits: reproducible, version-controlled, auditable deployments
- Security: review IaC for misconfigs before deploying (tools: Checkov, tfsec)

## Benefits of Automation

| Benefit | Description |
|---|---|
| **Efficiency gains** | Tasks that took hours done in seconds; analysts focus on higher-value work |
| **Consistent baselines** | Same configuration applied every time; human variation eliminated |
| **Standard infrastructure configs** | IaC ensures every environment is identical and secure |
| **Secure scaling** | Auto-scaling cloud resources automatically inherit security controls |
| **Employee retention** | Removing toil (repetitive manual work) improves analyst job satisfaction |
| **Faster reaction time** | Sub-second automated responses vs. minutes/hours for manual |
| **Workforce multiplier** | Small team can manage larger environment; do more with same staff |

---

## Considerations and Risks

| Consideration | Description | Mitigation |
|---|---|---|
| **Complexity** | Automation systems have many moving parts; harder to troubleshoot | Document thoroughly; test in staging first |
| **Cost** | SOAR platforms and implementation are expensive | Build business case with ROI; start with highest-value use cases |
| **Single point of failure** | If automation system fails, manual processes must be ready | Build redundancy; maintain manual runbooks |
| **Technical debt** | Complex automations become difficult to maintain over time | Regular reviews; keep automations simple and modular |
| **Ongoing supportability** | Requires dedicated staff with automation expertise | Document all runbooks; cross-train team |
| **Over-automation risk** | Automated response can take incorrect action (block legitimate traffic) | Use human-in-the-loop for high-impact actions |
| **Integration challenges** | Tools must have APIs; legacy tools may not support automation | Evaluate API support before purchasing tools |

## Automation in the Security Lifecycle

| Phase | Automation Examples |
|---|---|
| **Prevention** | Auto-apply patches, enforce config baselines, IaC deployments |
| **Detection** | SIEM correlation rules, automated threat hunting queries |
| **Response** | SOAR playbooks: block IP, isolate host, reset credentials |
| **Recovery** | Automated failover, backup restoration scripts |
| **Compliance** | Auto-generate compliance reports, continuous compliance scanning |

## Exam Tips

- **SOAR = Orchestration + Automation + Response** — it ties together multiple security tools via APIs.
- The key benefit of SOAR is **speed of response** — sub-second automated containment vs. manual minutes.
- **Playbook** = documented (manual) procedure; **Runbook** = automated version executed by SOAR.
- **Guard rails** prevent misconfigurations from persisting — they auto-remediate, not just alert.
- **IaC** ensures consistent, repeatable, auditable infrastructure deployments — security benefit is standardization.
- **CI/CD pipeline security** = "shift left" — find vulnerabilities during development, not after deployment.
- **Single point of failure** is the most significant risk of automation — if SOAR goes down, manual processes must be ready.
- **Employee retention** is a real benefit — removing toil reduces analyst burnout; Security+ explicitly lists this.
- **Complexity and technical debt** are the primary ongoing operational risks of automation.
- Automation is a **workforce multiplier** — doesn't replace analysts; enables them to handle more threats.
- Know the word **"orchestration"** — it's the coordination of multiple tools; automation is the execution of individual tasks.
