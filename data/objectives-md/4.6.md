# 4.6 — Identity and Access Management

## What you need to know

Identity and Access Management (IAM) ensures the right people have the right access to the right resources at the right time. This is one of the most heavily tested Domain 4 objectives. You need to know provisioning workflows, access control models (MAC/DAC/RBAC/ABAC), SSO protocols (LDAP/SAML/OAuth), MFA types, password security policies, and Privileged Access Management (PAM). Expect scenarios asking which access control model applies or which MFA factor is being described.

---

## User Provisioning and De-provisioning

### Joiner-Mover-Leaver (JML) Process

| Stage | Event | Security Action |
|---|---|---|
| **Joiner** | New employee hired | Create account; assign minimum necessary permissions; provision MFA |
| **Mover** | Employee changes roles/department | Review and modify access; remove old role permissions; add new ones |
| **Leaver** | Employee resigns or is terminated | **Immediately** disable account; revoke credentials; collect devices |

- **Account disabling vs. deletion** — disable first (preserve audit trail), delete after retention period
- **Access reviews** — periodic re-certification that users still need their current access; managers certify
- **Orphaned accounts** — accounts that remain active after the user has left; major security risk

### Identity Proofing
- The process of **verifying that someone is who they claim to be** before issuing credentials
- Methods: government ID, knowledge-based questions, biometric enrollment, in-person verification
- Required before issuing certificates, privileged accounts, or sensitive access

## Permission Assignment Principles

| Principle | Description |
|---|---|
| **Least privilege** | Grant only the minimum access required to perform the job function |
| **Need-to-know** | Even with appropriate clearance, access only to specific data needed for a task |
| **Separation of duties** | No single person can complete a critical process end-to-end |
| **Mandatory vacation** | Forced leave detects fraud (someone must cover for them) |
| **Job rotation** | Rotate roles to detect fraud and cross-train |

## Access Control Models

| Model | Full Name | Who Controls Access | How It Works | Use Case |
|---|---|---|---|---|
| **MAC** | Mandatory Access Control | System (policy-enforced) | Security labels on subjects and objects; must match | Military, government, classified systems |
| **DAC** | Discretionary Access Control | Resource owner | Owner sets permissions on their own resources | Traditional OS file systems (Windows, Linux) |
| **RBAC** | Role-Based Access Control | Administrator (via roles) | Permissions assigned to roles; users assigned to roles | Most corporate environments |
| **Rule-based** | Rule-Based Access Control | Administrator (via rules) | Conditions must be met (time of day, IP address) | Firewall ACLs, time-of-day restrictions |
| **ABAC** | Attribute-Based Access Control | Policy engine | Complex policies using multiple attributes | Cloud IAM, Zero Trust |

- **MAC** — labels like Top Secret, Secret, Confidential; access requires both clearance AND need-to-know
- **DAC** — file permissions in NTFS or Linux (`chmod`) are DAC; owner can share their files
- **RBAC** — "Help Desk" role gets ticket system access; no individual permission management
- **ABAC** — policy: "Allow access IF user.department=Finance AND resource.type=Budget AND time=BusinessHours"
- **Time-of-day restrictions** — Rule-based control; users can only access systems during defined hours

## Federation and SSO

### Federation
- Establishes **trust between identity systems** across different organizations
- Users authenticate with their home organization's IdP; trusted by the partner's SP
- Example: login to a partner portal using your company credentials
- Technologies: **SAML**, **OAuth + OIDC**, **WS-Federation**

### SSO (Single Sign-On)
- User authenticates **once** and gets access to multiple systems without re-authenticating

### Key SSO Protocols

| Protocol | Type | Purpose | Notes |
|---|---|---|---|
| **LDAP** | Directory protocol | Authentication and directory queries | Port 389 (unencrypted), 636 (LDAPS); active directory backend |
| **SAML** | XML-based standard | Web SSO; exchange authentication assertions | Federated identity; SP and IdP; used in enterprise SSO |
| **OAuth 2.0** | Authorization framework | Delegated access (not authentication) | Issues access tokens; grants third-party access without sharing credentials |
| **OpenID Connect (OIDC)** | Authentication layer on OAuth | Authentication (who you are) | Built on OAuth 2.0; adds ID token (JWT) |
| **Kerberos** | Ticket-based protocol | LAN authentication (Windows AD) | Uses tickets (TGT/service tickets); does not transmit passwords |

- **OAuth** is for **authorization** (what you can do), NOT authentication (who you are) — common exam trap
- **SAML** uses **assertions** (XML documents) to pass authentication data between IdP and SP
- **OIDC** adds authentication to OAuth — it IS used for authentication (adds the ID token)

### Attestation
- **Manager certifies** that their employee still needs their current access
- Periodic access review process — typically quarterly or annually
- Required by compliance frameworks (SOX, HIPAA, PCI DSS)
- Unreviewed access gets automatically revoked in strict implementations

## Multi-Factor Authentication (MFA)

MFA requires two or more factors from different categories.

### Authentication Factor Categories

| Factor | Category | Examples |
|---|---|---|
| **Something you know** | Knowledge | Password, PIN, security question, passphrase |
| **Something you have** | Possession | Hard token (RSA SecurID), smart card, soft token app, SMS code |
| **Something you are** | Inherence (Biometric) | Fingerprint, retina scan, facial recognition, voice |
| **Somewhere you are** | Location | GPS location, IP address geolocation |

### MFA Implementation Types

| Type | Description | Notes |
|---|---|---|
| **TOTP** | Time-based One-Time Password | Changes every 30 seconds; RFC 6238; Google Authenticator, Authy |
| **HOTP** | HMAC-based One-Time Password | Counter-based; each use increments counter |
| **Hard token** | Physical device generating OTP | RSA SecurID; most secure possession factor |
| **Soft token** | App-based OTP generator | Authenticator apps |
| **SMS code** | One-time code via text message | Least secure MFA; SIM swapping risk |
| **Push notification** | Approve/deny via mobile app | Microsoft Authenticator, Duo; phishing-resistant versions available |
| **Biometric** | Fingerprint, face, retina | Something you are; false accept/reject rates matter |
| **FIDO2 / Passkeys** | Passwordless public-key cryptography | Phishing-resistant; hardware-bound key |

- **Biometrics** measure: FAR (False Acceptance Rate), FRR (False Rejection Rate), CER/EER (Crossover Error Rate — where FAR = FRR; lower is better)
- **SMS is the weakest MFA** but still better than no MFA
- **FIDO2/passkeys** are the most phishing-resistant — private key never leaves the device

## Password Concepts

| Concept | Description | Notes |
|---|---|---|
| **Length** | Minimum number of characters | More important than complexity; 12+ recommended |
| **Complexity** | Mix of character types | Uppercase, lowercase, numbers, symbols |
| **Reuse prevention** | Password history prevents recent passwords | Enforce history of last 10-24 passwords |
| **Expiration** | Force periodic password changes | NIST SP 800-63B: **don't force expire unless compromised** |
| **Minimum age** | Minimum days before password can be changed | Prevents users from cycling through history quickly |
| **Account lockout** | Lock after N failed attempts | Prevents brute force; set lockout threshold carefully |
| **Password managers** | Encrypted vault stores unique passwords | Allows long, complex, unique passwords per site |
| **Passwordless** | FIDO2, passkeys, biometric | Eliminates passwords entirely |

- **NIST's current guidance** — do NOT require periodic expiration unless compromise is suspected; DO require long passwords; check against breached password lists

## Privileged Access Management (PAM)

| Concept | Description |
|---|---|
| **PAM** | Controls and monitors privileged (admin) accounts and access |
| **Just-in-time (JIT) access** | Grant elevated access only when needed; auto-revoke after time period |
| **Ephemeral credentials** | Temporary credentials that automatically expire |
| **Session recording** | Record all privileged sessions for audit and forensics |
| **Password vaulting** | PAM stores and rotates privileged account passwords; users never know the password |
| **Dual control** | Two people required to access the most sensitive systems |

- PAM solutions: **CyberArk**, **BeyondTrust**, **Delinea (formerly Thycotic)**
- **Privileged accounts** are high-value targets — compromise of one can lead to full domain compromise
- JIT access eliminates standing privileges — drastically reduces the attack surface

## Exam Tips

- **OAuth is NOT authentication** — it's authorization/delegation. Use **OIDC** or **SAML** for authentication.
- **SAML** uses XML **assertions** — common in enterprise web SSO and federation.
- **LDAP port 389** (unencrypted); **LDAPS port 636** (encrypted) — know both.
- **MAC** = system-enforced labels; **DAC** = owner decides; **RBAC** = roles; **ABAC** = attributes.
- **RBAC** is the most common model in corporate environments — permissions flow through roles.
- **TOTP changes every 30 seconds**; **HOTP is counter-based** (changes with each use).
- **SMS is the weakest MFA** — susceptible to SIM swapping and SS7 attacks.
- **FIDO2/passkeys** are the strongest and most phishing-resistant authentication method.
- **Least privilege** is applied at provisioning; **access reviews** (attestation) verify it's maintained over time.
- **PAM JIT access** = no standing privileges — access granted on demand and auto-revoked.
- Biometric **CER/EER** = the point where false accepts equal false rejects — lower CER = better accuracy.
- The **Joiner-Mover-Leaver** process ensures access is properly managed throughout employment lifecycle.
