# 4.1 — Security Techniques for Computing Resources

## What you need to know

Securing computing resources involves applying consistent, documented standards across every type of system in the organization. The exam tests your ability to identify the correct hardening technique for a given target — mobile devices, workstations, switches, routers, cloud infrastructure, servers, ICS/SCADA, and embedded systems. You also need to know wireless security standards, mobile device management approaches, application security testing methods, and the purpose of sandboxing. Expect scenario-based questions that ask which technique should be applied given a specific environment or threat.

---

## Secure Baselines

A **secure baseline** is a minimum-security configuration standard that every system must meet before deployment and must maintain throughout its lifecycle.

| Phase | Description | Example Actions |
|---|---|---|
| **Establish** | Define the standard configuration | Write policies, reference CIS Benchmarks, STIG |
| **Deploy** | Apply the baseline to new systems | Imaging, configuration management tools (Ansible, Chef, Puppet) |
| **Maintain** | Continuously verify and re-enforce baseline | Drift detection, periodic audits, automated compliance scans |

- **Configuration drift** occurs when a system's settings deviate from the established baseline over time
- Tools like **SCAP**, **CIS-CAT**, and **SIEM** can detect drift automatically
- **Gold images** are pre-hardened OS templates used for consistent, repeatable deployments

## Hardening Targets

### Mobile Devices
- Enable **full-disk encryption** and require a strong PIN/biometric
- Enforce **remote wipe** capability via MDM
- Disable unused services: Bluetooth, NFC, hotspot
- Apply **app whitelisting** — only approved apps allowed
- Enable **screen lock timeout** and failed-login wipe policies
- Keep OS **patched and updated**

### Workstations
- Remove unnecessary software (reduce attack surface)
- Disable autorun/autoplay on removable media
- Enable **host-based firewall** and **antivirus/EDR**
- Apply **Group Policy** settings (Windows) — screen lock, USB restriction
- Require **MFA** for login
- Use **standard user accounts** — not local admin for daily use

### Switches
- Disable unused ports (prevents unauthorized network access)
- Enable **port security** — limit MAC addresses per port
- Use **VLAN segmentation** to isolate traffic
- Disable **CDP/LLDP** on external-facing ports (prevents topology leakage)
- Enable **BPDU Guard** (protects Spanning Tree from attacks)
- Use **802.1X** for port-based Network Access Control

### Routers
- Change default credentials immediately
- Disable unnecessary services (HTTP management interface, Telnet)
- Use **SSH** instead of Telnet for management
- Apply **ACLs** to restrict management plane access
- Enable **logging** and forward to SIEM
- Update firmware regularly

### Cloud Infrastructure
- Use **Identity and Access Management (IAM)** — principle of least privilege
- Enable **multi-factor authentication** on all cloud accounts
- Encrypt data at rest and in transit (S3 encryption, KMS)
- Use **Security Groups** and network ACLs to restrict traffic
- Enable **CloudTrail / audit logging** — track all API calls
- Implement **Cloud Security Posture Management (CSPM)** to detect misconfigurations

### Servers
- Remove or disable unused roles and features
- Apply OS hardening (disable unused ports, services)
- Use **dedicated service accounts** with minimal privileges
- Enable **NTP** synchronization for accurate log timestamps
- Apply **patches** — OS, middleware, application
- Implement **file integrity monitoring (FIM)**

### ICS/SCADA (Industrial Control Systems)
- **Air gap** — physically isolate from corporate/internet networks when possible
- If connected, use **strict firewall rules** and DMZ between corporate and OT network
- Apply **allowlisting** — only known-good executables permitted
- Patching is difficult (uptime constraints) — use **compensating controls**
- Monitor for anomalous traffic (ICS-specific protocols: Modbus, DNP3)
- **Purdue Model** — hierarchical segmentation of OT/IT networks

### Embedded Systems / IoT
- Change **default credentials** immediately
- Disable unused network services
- Isolate on **dedicated VLAN** or network segment
- Enable firmware updates where possible
- Apply **network-level controls** since device hardening is often limited

## Wireless Security

| Standard | Protocol | Encryption | Notes |
|---|---|---|---|
| **WPA2** | 802.11i | **CCMP/AES** | Current minimum standard; TKIP deprecated |
| **WPA3** | 802.11i++ | GCMP-256 (AES) | Preferred; uses SAE instead of PSK |
| **WPA** | 802.11i (partial) | TKIP | Deprecated — do not use |
| **WEP** | Legacy | RC4 | Broken — never use |

- **CCMP** (Counter Mode with Cipher Block Chaining Message Authentication Code Protocol) — uses **AES** encryption; used in WPA2
- **TKIP** (Temporal Key Integrity Protocol) — stopgap fix for WEP; now deprecated and vulnerable
- **SAE** (Simultaneous Authentication of Equals) — replaces PSK in WPA3; resistant to offline dictionary attacks
- **WPA3-Enterprise** uses 192-bit minimum security (GCMP-256, HMAC-SHA-384)
- **Open networks** with WPA3 use **Opportunistic Wireless Encryption (OWE)** — encrypts even without a password

## Mobile Solutions

### MDM / Mobile Device Management

| Feature | Description |
|---|---|
| **MDM** | Centrally manage mobile devices — push policies, apps, wipe |
| **MAM** | Mobile Application Management — manage apps only, not device |
| **UEM** | Unified Endpoint Management — manages all endpoint types |
| **Remote wipe** | Erase device data if lost/stolen |
| **Geofencing** | Restrict features based on location |
| **Containerization** | Separate corporate and personal data on same device |

### Deployment Models

| Model | Stands For | Who Owns Device | Notes |
|---|---|---|---|
| **BYOD** | Bring Your Own Device | Employee | Flexibility but less control; privacy concerns |
| **COPE** | Corporate-Owned, Personally Enabled | Employer | More control; employee can use for personal |
| **CYOD** | Choose Your Own Device | Employer (from approved list) | Employee picks from approved options |
| **COBO** | Corporate-Owned, Business Only | Employer | Maximum control; no personal use |

### Connection Methods
- **Cellular** (4G/5G) — native carrier connection; less dependent on corporate network
- **Wi-Fi** — connect to wireless; ensure VPN usage on untrusted networks
- **Bluetooth** — short range; disable when not needed; **bluejacking/bluesnarfing** risks
- **NFC** — near field; payment apps; tap to pair; disable when not needed
- **USB** — tethering, charging, data transfer; disable in MDM if not needed
- **VPN** — always-on VPN tunnels corporate traffic; protects on untrusted networks

## Application Security

| Technique | Description | When Used |
|---|---|---|
| **Input validation** | Validate all user input server-side; reject unexpected data | Prevents SQLi, XSS, buffer overflow |
| **Secure cookies** | HttpOnly (no JS access), Secure (HTTPS only), SameSite (CSRF protection) | Web application session security |
| **SAST** | Static Application Security Testing — analyze source code without execution | During development (shift left) |
| **DAST** | Dynamic Application Security Testing — test running application | During testing / staging |
| **Code signing** | Developer digitally signs code; OS verifies signature before execution | Ensures integrity and authenticity of software |
| **Obfuscation** | Make code harder to reverse-engineer | Protects proprietary logic; limits attacker reconnaissance |

- **SAST** is "white box" — has access to source code; finds issues early; cannot find runtime vulnerabilities
- **DAST** is "black box" — tests the running app like an attacker; finds runtime, config, and integration issues
- **IAST** (Interactive Application Security Testing) — hybrid; instruments the app from inside while DAST runs
- Input validation must happen **server-side** — client-side validation is easily bypassed

## Sandboxing

- **Sandboxing** isolates code or processes in a restricted environment to prevent them from affecting the host system
- Used for: malware analysis, testing untrusted software, browser tab isolation, email attachment scanning
- Examples: **Windows Sandbox**, browser isolation, **Cuckoo Sandbox** (malware analysis), containers
- Benefit: if the sandboxed code is malicious, the damage is **contained**
- Antivirus/EDR tools use sandboxing to **detonate** suspicious files safely

## Exam Tips

- **WPA3 uses SAE** — not PSK. SAE prevents offline dictionary attacks. This is a key differentiator.
- **TKIP is deprecated** — if an exam option says "use TKIP for compatibility," that is wrong.
- **CCMP/AES** is used in WPA2; know the pairing cold.
- For ICS/SCADA, the preferred solution is **air gap** — if that's not possible, use strict DMZ segmentation.
- **SAST** = code analysis without running (white box, development phase). **DAST** = testing the running app (black box, testing phase).
- **BYOD** provides the **least organizational control** — MDM helps, but employee-owned devices are harder to fully manage.
- **COPE** gives the most control while still allowing personal use.
- **Input validation must be server-side** — client-side validation is not a security control.
- **Code signing** provides integrity and authenticity — ensures software hasn't been tampered with.
- Hardening always means **reducing attack surface** — disable unused services, change defaults, apply least privilege.
- **Sandboxing** contains the blast radius — malicious code cannot escape to the host.
- For embedded/IoT devices: **network segmentation is the primary control** because device-level hardening is limited.
