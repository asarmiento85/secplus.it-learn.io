# 4.5 — Modifying Enterprise Capabilities for Security

## What you need to know

This objective covers the security technologies deployed at the enterprise level to detect, prevent, and respond to threats. It is one of the broadest objectives — spanning firewalls, IDS/IPS, web filters, OS security, secure protocols, email security, DLP, FIM, NAC, EDR/XDR, and user behavior analytics. The exam will present scenarios and ask which tool or protocol is appropriate. Know each tool's purpose, where it sits in the network, and how it differs from similar tools.

---

## Firewall Configuration

| Concept | Description |
|---|---|
| **ACLs / Rules** | Ordered list of permit/deny rules based on source IP, dest IP, port, protocol |
| **Allow rules** | Explicitly permit matching traffic |
| **Deny rules** | Block matching traffic; implicit deny at the end of every rule set |
| **Implicit deny** | Default: if no rule matches, traffic is denied |
| **Stateful inspection** | Tracks connection state; allows return traffic automatically |
| **Stateless filtering** | Evaluates each packet independently; no connection tracking |
| **Screened subnet / DMZ** | Network zone between internet and internal network; hosts public-facing servers |

- **Next-Generation Firewall (NGFW)** — adds application awareness, user identity, SSL inspection, IPS to traditional firewall
- **Web Application Firewall (WAF)** — specifically protects web apps; inspects HTTP/HTTPS; blocks SQLi, XSS
- **Unified Threat Management (UTM)** — all-in-one device: firewall + IPS + AV + VPN + web filter

## IDS and IPS

| Feature | IDS | IPS |
|---|---|---|
| **Full name** | Intrusion Detection System | Intrusion Prevention System |
| **Action** | Detects and alerts | Detects, alerts, AND blocks |
| **Placement** | Out-of-band (tap/span port) | Inline (traffic must pass through) |
| **Control type** | Detective | Preventive |
| **Risk** | Generates false positives (alert noise) | False positives cause traffic drops (worse impact) |

### Detection Methods

| Method | Description | Pros/Cons |
|---|---|---|
| **Signature-based** | Matches traffic against database of known attack patterns | Low false positives; misses zero-days |
| **Anomaly-based** | Deviates from established traffic baseline | Detects zero-days; higher false positives |
| **Behavior-based** | Analyzes behavior patterns | Similar to anomaly; applied to endpoints |
| **Heuristic** | Rules-based + machine learning to identify suspicious behavior | Good for new variants; complex tuning |

### Deployment Scope

| Type | Monitors | Abbreviation |
|---|---|---|
| **Network-based** | Network traffic | NIDS / NIPS |
| **Host-based** | Activity on a single host | HIDS / HIPS |

## Web Filter

- **Purpose**: Control which websites users can access; block malicious or non-business content
- **Agent-based**: Software installed on device; filters regardless of location (remote workers)
- **Centralized proxy**: All traffic routes through proxy server; one control point
- **URL scanning**: Examine destination URLs against blocklists and reputation databases
- **Content categorization**: Classify URLs by category (social media, gambling, news); block/allow by category
- **Block rules**: Explicit rules to block specific domains, IPs, or URL patterns
- **Reputation scoring**: Sites scored 1-100 based on threat intelligence; low score = blocked
- **DNS filtering**: Block at DNS resolution level (Cisco Umbrella, Cloudflare Gateway) — simplest form of web filtering

## OS Security

### Group Policy Objects (GPO) — Windows
- **GPO** = policy container applied to users/computers in Active Directory
- Can enforce: screen lock timeout, password policies, software restrictions, USB disabling, firewall settings
- **Default Domain Policy** applies to all users/computers
- **OU-level GPOs** apply to specific organizational units
- **Group Policy Preferences** — configure settings that users can override (less restrictive)

### SELinux (Security-Enhanced Linux)
- **Mandatory Access Control (MAC)** system built into Linux kernel
- Every process and file has a **security label** (context)
- Access decisions based on policy, not just file permissions
- Modes: **Enforcing** (policies enforced), **Permissive** (logs but doesn't block), **Disabled**
- Default in Red Hat/CentOS; key differentiator for government/DoD Linux deployments

## Secure Protocols

### Network and Remote Access

| Protocol | Purpose | Replaces | Port(s) |
|---|---|---|---|
| **SSH** | Encrypted remote administration | Telnet (23) | 22 |
| **TLS 1.2 / 1.3** | Encrypted transport for web, email, other protocols | SSL | 443 (HTTPS) |
| **HTTPS** | HTTP over TLS; secure web browsing | HTTP | 443 |
| **IPSec** | Encrypts IP packets; used in VPNs | Unencrypted IP | N/A |
| **SNMPv3** | Authenticated/encrypted network device management | SNMPv1/v2 | 161/162 |

### File Transfer

| Protocol | Description | Port |
|---|---|---|
| **SFTP** | SSH-based file transfer; uses SSH for encryption | 22 |
| **FTPS** | FTP over TLS; explicit or implicit TLS | 990 (implicit), 21 (explicit) |
| **SCP** | Secure Copy Protocol; uses SSH | 22 |

### Directory and Email

| Protocol | Description | Port |
|---|---|---|
| **LDAP** | Directory lookups (unencrypted) | 389 |
| **LDAPS** | LDAP over TLS (encrypted) | 636 |
| **S/MIME** | Email encryption and digital signing | N/A (email standard) |
| **SRTP** | Secure Real-time Transport Protocol; encrypts VoIP | N/A |

### DNS Security

| Protocol | Description |
|---|---|
| **DNSSEC** | Cryptographically signs DNS records; prevents DNS spoofing |
| **DNS over HTTPS (DoH)** | DNS queries sent over HTTPS; prevents interception |
| **DNS over TLS (DoT)** | DNS queries sent over TLS; port 853 |

### IPSec Components

| Component | Description |
|---|---|
| **AH** (Authentication Header) | Provides integrity and authentication; NO encryption |
| **ESP** (Encapsulating Security Payload) | Provides integrity, authentication, AND encryption |
| **Transport mode** | Encrypts only the payload; IP header intact; used host-to-host |
| **Tunnel mode** | Encrypts entire original packet; new IP header added; used in VPNs (site-to-site) |

## Email Security

| Technology | Purpose | How It Works |
|---|---|---|
| **SPF** | Authorize which servers can send email for a domain | DNS TXT record listing authorized IP ranges |
| **DKIM** | Sign outgoing email to prove it's from the domain | Private key signs message; public key in DNS for verification |
| **DMARC** | Policy for what to do when SPF/DKIM fail | DNS record: none (report), quarantine, reject |
| **Secure gateway** | Scan inbound and outbound email | Filters spam, phishing, malware, DLP enforcement |

- **SPF** prevents email spoofing at the sending server level
- **DKIM** provides email integrity — ensures message wasn't modified in transit
- **DMARC** ties SPF and DKIM together and enforces a policy on failures
- **S/MIME** provides end-to-end email encryption and signing using certificates

## Other Enterprise Security Capabilities

### FIM (File Integrity Monitoring)
- Monitors files and system directories for unauthorized changes
- Creates **hash values** (checksums) of files; alerts when hash changes
- Tool: **Tripwire** (most commonly tested)
- Use cases: detect malware installation, config changes, unauthorized modifications to critical files

### DLP (Data Loss Prevention)

| DLP Type | Where It Monitors |
|---|---|
| **Network DLP** | Traffic at network perimeter |
| **Endpoint DLP** | Activity on individual devices (USB, print, email) |
| **Cloud DLP** | Data in cloud storage and SaaS applications |

### NAC (Network Access Control)
- Controls **which devices can connect to the network** based on compliance and identity
- **802.1X** — port-based NAC; requires authentication before network access is granted
- **Health checks** — verify device has AV, updated patches, correct config before granting access
- **Remediation VLAN** — non-compliant devices placed in restricted network for remediation
- Agents (persistent or dissolvable) check device health posture

### EDR and XDR

| Technology | Scope | Key Capabilities |
|---|---|---|
| **EDR** | Endpoint | Behavioral detection, threat hunting, automated response, telemetry collection |
| **XDR** | Extended (endpoint + network + cloud + email) | Correlates telemetry across all domains; single console |

- **EDR** = evolution of antivirus; behavioral analysis, not just signatures
- **XDR** = EDR extended to all telemetry sources; improves detection by correlating across domains

### UBA (User Behavior Analytics)
- Establishes **baselines** of normal user behavior
- Alerts on deviations: unusual login times, accessing unusual files, large data transfers
- Primarily detects **insider threats** and compromised accounts
- **UEBA** (User and Entity Behavior Analytics) — extends UBA to devices and services

## Exam Tips

- **IDS = Detective; IPS = Preventive** — IPS is inline and can block; IDS is passive and only alerts.
- **HIDS/HIPS** monitors a single host; **NIDS/NIPS** monitors network traffic.
- **Signature-based** detection misses zero-days; **anomaly-based** detects zero-days but has more false positives.
- **ESP provides encryption; AH does not** — for confidentiality, always use ESP.
- **IPSec tunnel mode** encapsulates the entire original packet — used for site-to-site VPNs.
- **SFTP uses SSH (port 22)**; FTPS uses TLS over port 990/21 — these are completely different protocols.
- **LDAP uses port 389 (unencrypted); LDAPS uses port 636 (encrypted)** — know both ports.
- **DMARC** is the policy that says what to do when SPF/DKIM fail — reject, quarantine, or none.
- **FIM** detects changes to files by comparing hashes — **Tripwire** is the classic example.
- **NAC with 802.1X** authenticates devices/users before granting port access.
- **UBA detects insider threats** by identifying deviations from established behavioral baselines.
- **XDR** extends EDR to correlate telemetry across endpoints, network, cloud, and email.
