# 1.3 — Change Management Processes

## What you need to know

Change management is a formal process for controlling modifications to IT systems and infrastructure. From a security perspective, uncontrolled changes are one of the most common sources of vulnerabilities — a misconfigured firewall rule, an untested patch, or an undocumented dependency change can introduce serious security gaps. The Security+ exam tests both the business process side (how changes are approved and tracked) and the technical implications (what actually changes on the systems).

---

## Why Change Management Matters for Security

- **Uncontrolled changes = vulnerabilities** — a change made outside the process may not be tested, documented, or reversible
- **Accountability** — change records establish who made what change and when
- **Auditability** — regulators and auditors require evidence that changes were controlled
- **Availability protection** — changes during business hours without testing cause outages
- **Configuration drift** — systems gradually diverge from their secure baseline when changes are untracked

---

## Business Process Components

These are the administrative and procedural elements of a change management process.

| Component | Description | Security Significance |
|---|---|---|
| **Approval Process / CAB** | Change Advisory Board reviews and approves or rejects change requests | Prevents unauthorized or risky changes from being implemented |
| **Ownership** | Designated owner responsible for the change | Single point of accountability; owner is responsible if something goes wrong |
| **Stakeholders** | All parties affected by the change | Ensures impacted teams are notified and can prepare |
| **Impact Analysis** | Assessment of what could go wrong; risk to systems and data | Identifies security risks before the change is made |
| **Test Results** | Evidence that the change was tested in a lab/staging environment | Verifies the change works and doesn't break other systems |
| **Backout Plan** | Documented steps to undo the change if something goes wrong | Ensures availability can be restored quickly |
| **Maintenance Window** | Scheduled time period when changes are allowed | Reduces risk to availability; changes happen during low-impact periods |
| **Standard Operating Procedure (SOP)** | Documented step-by-step instructions for implementing the change | Ensures consistency; prevents mistakes from memory lapses |

### Change Advisory Board (CAB)
- Reviews all significant change requests before approval
- Members typically include: IT managers, security team, business owners, operations
- Emergency changes may bypass CAB with expedited approval (e-CAB)
- Change requests are submitted as **RFCs (Requests for Change)**

### Types of Changes
| Type | Description | Example |
|---|---|---|
| **Standard** | Pre-approved, low-risk, routine | Adding a user account, routine patch |
| **Normal** | Goes through full CAB approval process | New server deployment, firewall rule change |
| **Emergency** | Urgent; expedited approval; documented after the fact | Critical security patch for active exploit |

## Technical Implications

These are the technical security considerations when a change is implemented.

### Allow Lists and Deny Lists
- **Allow list (whitelist)** — only explicitly approved applications/traffic are permitted; everything else is blocked
- **Deny list (blacklist)** — explicitly blocked applications/traffic; everything else is permitted
- **Change implication:** Adding software requires updating allow lists; blocking new threats requires updating deny lists
- Failure to update these lists = new software may be blocked, or malicious software may run undetected

### Restricted Activities
- Certain activities are prohibited during change windows or without approval
- Examples: changes to production systems without CAB approval, disabling security controls, bypassing MFA
- **Separation of duties** — the person who writes code should not be the person who deploys it to production

### Downtime
- **Planned downtime** — scheduled maintenance window; stakeholders notified in advance
- **Unplanned downtime** — result of a change gone wrong; no backout plan = longer outage
- Downtime planning includes estimating impact and duration; communicating to affected users

### Service Restarts
- Many changes require services or systems to be restarted to take effect
- Example: patching an OS may require a reboot; changing a web server config requires a service restart
- **Security implication:** Service restarts can cause brief unavailability; must be planned

### Legacy Applications
- Older applications may **break** when underlying systems are patched or updated
- Legacy systems often cannot be patched (the vendor no longer supports them)
- **Security implication:** Legacy systems may require compensating controls (network segmentation, enhanced monitoring) instead of patching
- Dependencies on legacy apps must be identified in impact analysis

### Dependencies
- Modern systems have complex interdependencies — changing one system can break others
- **Application dependencies:** App A relies on Library B which relies on Service C
- **Network dependencies:** Changing a firewall rule can block traffic that other systems need
- Impact analysis must map all dependencies before a change is approved

## Documentation Requirements

Change management generates and requires documentation at every stage.

### Updating Diagrams
- **Network topology diagrams** — must be updated when new devices are added, removed, or reconfigured
- **Data flow diagrams** — updated when data paths change (new service, new integration)
- **Asset inventories** — updated when hardware/software is added or removed
- Outdated diagrams are a security risk — they lead to inaccurate risk assessments and missed dependencies

### Updating Policies and Procedures
- **Security policies** — may need updating if the change creates new requirements
- **Incident response plans** — updated when new systems are added (new procedures may be needed)
- **Runbooks and SOPs** — operational documents updated to reflect new configurations

## Version Control

Version control tracks changes to code, configuration files, and documentation over time.

### Security Benefits of Version Control
- **Rollback capability** — can revert to a known-good configuration if a change causes problems
- **Audit trail** — every change is logged with who made it, what changed, and when
- **Unauthorized change detection** — differences between expected and actual config reveal tampering
- **Collaboration control** — review and approval process before changes merge to production

### Version Control in Configuration Management
- **Configuration baselines** — a documented, approved configuration that all systems should match
- **Configuration Management Database (CMDB)** — inventory of all IT assets and their configurations
- **Drift detection** — automated tools compare actual configs to baseline; alert on differences
- **Infrastructure as Code (IaC)** — configurations stored as version-controlled code (Terraform, Ansible, Puppet)

### Common Version Control Systems
- **Git** — most widely used; distributed version control
- **SVN (Subversion)** — centralized version control
- **Branching** — work on changes in isolation before merging to main branch

## Change Management and Security Controls

| Scenario | Change Management Action |
|---|---|
| Applying a critical patch | Test in staging → CAB approval (or e-CAB for emergency) → maintenance window → document results |
| Adding new firewall rule | Impact analysis → test in lab → CAB approval → implement in maintenance window → update diagrams |
| Deploying new application | Stakeholder notification → allow list update → dependency check → test → CAB approval → SOP documented |
| Retiring legacy system | Dependency mapping → stakeholder notification → update diagrams and CMDB → archive documentation |

## Exam Tips

- **CAB (Change Advisory Board)** is the approval body for significant changes — know this acronym.
- **Backout plan** = how you undo the change; required for all significant changes; exams often ask what is missing when a change goes wrong.
- **Maintenance window** = the scheduled time when changes are permitted; changes outside this window = change management violation.
- **Impact analysis** is performed *before* implementing a change — it identifies risks and dependencies.
- **Allow list / deny list** must be updated as part of change management; forgetting to update = security gap.
- **Legacy applications** create change management complications — they often cannot be updated and require compensating controls.
- **Version control** provides rollback AND audit trail — both are security benefits.
- **Separation of duties** in change management: the developer should not be the one deploying to production.
- **Documentation** is not optional — if a change isn't documented, it didn't happen from a compliance perspective.
- **Emergency changes** still require documentation — expedited approval, then documented after the fact.
- Watch for questions about what happens when change management is **bypassed** — the answer usually involves unauthorized changes, outages, or security breaches.
