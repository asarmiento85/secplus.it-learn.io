# 3.3 — Data Protection Concepts and Strategies

## What you need to know

Data protection is one of the most tested areas on the Security+ exam. You need to understand data types, classification levels, the three states of data, and the specific protection methods that apply to each. The exam will present scenario questions where you must select the appropriate control — for example, knowing when to use tokenization vs encryption vs masking, or identifying which regulation governs a specific data type.

---

## Data Types

| Data Type | Description | Governing Framework |
|-----------|-------------|-------------------|
| **Regulated** | Data subject to specific legal/compliance requirements | HIPAA, PCI-DSS, GDPR |
| **Trade secret** | Proprietary business information giving competitive advantage | NDA, trade secret law |
| **Intellectual property** | Creative works, inventions, brand identifiers | Patents, copyrights, trademarks |
| **Legal information** | Attorney-client privileged communications, litigation files | Attorney-client privilege |
| **Financial information** | Accounting records, banking data, payroll | SOX, PCI-DSS, GLBA |
| **Human-readable** | Text, documents, structured data readable without special tools | — |
| **Non-human-readable** | Binary files, encrypted data, compiled code, compressed archives | — |

### Key Regulatory Frameworks

| Regulation | Scope | Key Requirement |
|-----------|-------|----------------|
| **HIPAA** | Protected Health Information (PHI) in the US | Protect health data; breach notification within 60 days |
| **PCI-DSS** | Payment card data (CHD — cardholder data) | Encrypt cardholder data; tokenization common compliance tool |
| **GDPR** | Personal data of EU residents (anywhere in the world) | Right to erasure; data minimization; 72-hour breach notification |
| **GLBA** | Financial data held by financial institutions in the US | Safeguards rule; privacy notices |
| **SOX** | Financial reporting of US public companies | Integrity of financial records; access controls |

## Data Classification

Data classification systems define sensitivity levels and required handling for each level. The exam uses both government and corporate classification models.

### Government Classification (Most to Least Sensitive)

| Level | Description |
|-------|-------------|
| **Top Secret** | Disclosure would cause exceptionally grave damage to national security |
| **Secret** | Serious damage to national security if disclosed |
| **Confidential** | Damage to national security if disclosed |
| **Unclassified** | No classification restriction |

### Corporate Classification (Most to Least Sensitive)

| Level | Description | Example |
|-------|-------------|---------|
| **Restricted / Critical** | Most sensitive; severe impact if disclosed; highest protection required | Encryption keys, board decisions, M&A data |
| **Confidential / Sensitive** | Business impact if disclosed; limited distribution | Customer PII, source code, financial forecasts |
| **Private / Internal** | For company use only; not for public release | Internal policies, org charts, employee data |
| **Public** | Approved for public release; no harm if disclosed | Marketing materials, press releases |

- Data must be **labeled** (tags, headers, footers, metadata) so users know how to handle it
- Data **owners** are responsible for classification; data **custodians** implement the technical controls

## Data States and Required Protections

### Data at Rest

- **Definition**: data stored on a medium — disk, database, backup tape, USB drive, cloud storage
- **Risks**: physical theft of storage media, unauthorized database access, stolen backups
- **Protection methods**:
  - **Full disk encryption**: encrypts entire storage volume (BitLocker for Windows, FileVault for macOS, dm-crypt/LUKS for Linux)
  - **Database encryption**: Transparent Data Encryption (TDE) for SQL Server, Oracle, MySQL
  - **File/folder encryption**: encrypt specific sensitive files (EFS, PGP)
  - **AES-256**: the standard symmetric encryption algorithm for data at rest
  - **Backup encryption**: backups must be encrypted — a stolen unencrypted backup tape = data breach

### Data in Transit

- **Definition**: data moving across a network — LAN, WAN, internet, or between application components
- **Risks**: man-in-the-middle attacks, packet capture/sniffing, session hijacking
- **Protection methods**:
  - **TLS (Transport Layer Security)**: encrypts TCP-based communications; used by HTTPS, SMTPS, IMAPS
  - **VPN (IPsec or SSL/TLS)**: encrypts all traffic between endpoints or networks
  - **SFTP** (SSH File Transfer Protocol): encrypted file transfer; replaces FTP
  - **HTTPS**: HTTP over TLS; mandatory for any site handling sensitive data
  - **FTPS**: FTP over TLS (different from SFTP — uses different port and protocol mechanism)

### Data in Use

- **Definition**: data actively being processed in memory (RAM) — being read, computed on, or manipulated
- **Risks**: memory scraping, cold boot attacks, insecure process memory dumps
- **Protection methods**:
  - **Secure enclaves** (Intel SGX, AMD SEV): hardware-isolated memory regions for sensitive computations
  - **Memory encryption**: encrypt RAM contents so physical memory extraction is not useful
  - **Application-level controls**: minimize how long sensitive data remains in memory; zero out after use

## Data Sovereignty and Geolocation

- **Data sovereignty**: data is subject to the laws of the country where it physically resides or is processed
  - Example: EU personal data stored on a US server is subject to GDPR — the company must ensure US server meets GDPR standards, or use Standard Contractual Clauses (SCCs)
  - Example: data stored in China is subject to China's cybersecurity law; government may require access
- **Geolocation controls**: cloud providers allow you to restrict which geographic regions store and process your data
  - AWS: Region selection; data does not leave a region unless you configure it
  - Azure: Geo-replication pairs; data residency commitments
- Compliance implications: PCI-DSS, HIPAA, and GDPR all have data residency implications

## Data Protection Methods

### Encryption

- Transforms plaintext into ciphertext using a cryptographic algorithm and key
- **Most important data protection control** — protects confidentiality even if other controls fail
- Must be paired with **key management**: secure key storage (HSM), key rotation, key access control
- Encryption at rest + encryption in transit = defense in depth for data

### Hashing

- One-way transformation producing a fixed-length digest from input data
- Cannot be reversed (not encryption); used for **integrity verification**, not confidentiality
- If two inputs produce the same hash → **collision** (SHA-1 and MD5 are broken/deprecated)
- **SHA-256 / SHA-3**: current standards
- Passwords should be stored as hashed + salted values (bcrypt, Argon2, PBKDF2)

### Masking

- Hides part of sensitive data while preserving format and usability
- The real data exists and is preserved; only the displayed value is obscured
- Example: showing `****-****-****-1234` for a credit card number on a receipt
- Used in: customer service screens, logs, reports — anywhere where the full value is not needed
- Data masking is different from tokenization: masking shows a partial real value; tokenization replaces with a surrogate

### Tokenization

- Replaces sensitive data with a non-sensitive surrogate value (a **token**)
- A **token vault** (secure mapping database) maps tokens back to the real values
- The token has no mathematical relationship to the original — cannot be reversed without the vault
- Common use: **PCI-DSS compliance** — credit card numbers stored as tokens; only the payment processor holds the vault
- Reduces scope of PCI compliance — systems that only see tokens are out of PCI scope

| Feature | Masking | Tokenization | Encryption |
|---------|---------|-------------|-----------|
| Reversible | No (hides partial value) | Yes (via token vault) | Yes (via key) |
| Original preserved | Yes (somewhere) | Yes (in vault) | Yes (via decryption) |
| PCI use | Receipts/displays | Primary CHD storage | Data at rest/transit |
| Format preserving | Yes (partial) | Can be | Can be (FPE) |

### Obfuscation

- Making data harder to understand without fully encrypting or removing it
- Less rigorous than encryption — often used in code (obfuscated JavaScript) or database displays
- Example: replacing real names with pseudonyms in a test dataset
- Not a substitute for encryption; primarily reduces casual exposure risk

### Segmentation

- Isolates sensitive data to specific systems and network zones
- Example: cardholder data isolated in a separate network segment (PCI cardholder data environment / CDE)
- Reduces the blast radius of a breach — attacker who compromises one segment cannot directly reach the sensitive segment
- Network segmentation + data segmentation = defense in depth

### Permission Restrictions (Access Control)

- **Need-to-know**: users only access data required for their specific job function
- **Role-Based Access Control (RBAC)**: permissions assigned to roles; users inherit permissions from role
- **Attribute-Based Access Control (ABAC)**: permissions based on user attributes, data attributes, and environment (time, location)
- **Least privilege principle**: grant minimum permissions required; review and revoke regularly
- **Data Loss Prevention (DLP)**: monitors and controls data movement; blocks unauthorized transfer of sensitive data

## Data Lifecycle Management

| Phase | Security Concern |
|-------|-----------------|
| Creation / Collection | Classify at creation; only collect what is needed (data minimization) |
| Storage | Encrypt at rest; access controls; backup |
| Use | Access logging; DLP; secure memory handling |
| Sharing / Transmission | Encrypt in transit; authorized recipients only |
| Archival | Encrypted archives; retention policy compliance |
| Disposal / Destruction | Secure deletion (DoD 5220.22-M wiping, degaussing, physical destruction for drives) |

## Exam Tips

- Know the three data states and which protection applies: **at rest = encryption (AES-256, BitLocker, TDE)**, **in transit = TLS/VPN/SFTP/HTTPS**, **in use = secure enclaves/memory encryption**
- Tokenization vs masking: tokenization replaces data with a surrogate stored in a vault; masking shows partial real data without replacing it
- PCI-DSS protects payment card data; HIPAA protects PHI; GDPR protects EU personal data — know which regulation applies to which scenario
- Data sovereignty = where data physically lives determines which laws apply — GDPR follows EU resident data globally
- Hashing = integrity only; it is one-way and not encryption — do not confuse them
- Data owner = classifies and sets policy; data custodian = implements technical controls
- Tokenization is the primary mechanism for reducing PCI scope — systems that only see tokens are out of scope
- Secure deletion is required at end-of-life: wiping, degaussing, or physical destruction depending on media type
- GDPR requires breach notification within 72 hours; HIPAA requires notification within 60 days
- DLP enforces data protection policies by monitoring, detecting, and blocking unauthorized data movement
