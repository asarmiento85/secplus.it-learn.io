# 3.1 — Architecture Models and Security Implications

## What you need to know

The Security+ exam tests your ability to identify the security implications of different architecture models — from cloud service models to virtualization, IoT, and network design patterns. You need to understand not just what each model is, but what unique risks each introduces and how responsibility is shared between vendors and customers. Expect scenario questions asking you to identify the correct architecture type or the primary security risk it introduces.

---

## Cloud Service Models

| Model | Managed By Provider | Managed By Customer | Examples |
|-------|-------------------|-------------------|---------|
| **IaaS** | Hardware, hypervisor, networking, storage | OS, runtime, middleware, apps, data | AWS EC2, Azure VMs, Google Compute Engine |
| **PaaS** | Hardware through runtime/platform | Application code, data | AWS Elastic Beanstalk, Heroku, Google App Engine |
| **SaaS** | Everything (hardware through application) | Data and access management only | Office 365, Salesforce, Google Workspace |

### Shared Responsibility Matrix

- **CSP (Cloud Service Provider)** is always responsible for the physical infrastructure, network fabric, and hypervisor
- **Customer** is always responsible for their data, access management, and how they configure the services they consume
- The higher up the stack (IaaS to SaaS), the more the CSP takes on — and the less visibility the customer has
- IaaS: customer patches OS; PaaS: CSP patches platform, customer patches app; SaaS: CSP patches everything

### Hybrid Cloud

- Combines on-premises infrastructure with one or more cloud environments
- **Security challenge**: security policies, monitoring, and identity must span both environments consistently
- Data can flow between environments — increases exposure if not properly controlled
- Requires unified identity management (e.g., Azure AD hybrid sync) and consistent encryption policies across both sides

### Third-Party Vendors and Supply Chain Risk

- **Vendor lock-in**: deep dependence on a single provider limits ability to migrate if vendor is compromised or fails
- **Supply chain risk**: a compromise upstream (vendor, open-source library, firmware) propagates to all downstream customers
- Vet vendors with SOC 2 Type II audit reports, penetration test summaries, and SLAs
- **SolarWinds attack (2020)**: canonical supply chain attack — malicious update distributed to ~18,000 customers

## Infrastructure as Code (IaC)

- Provision and manage infrastructure using declarative configuration files rather than manual processes
- Tools: **Terraform** (multi-cloud), **AWS CloudFormation** (AWS-specific), **Azure Bicep** (Azure-specific)
- Benefits: consistent deployments, version-controlled infrastructure, auditable changes, rapid scaling and rollback

### IaC Security Risks

| Risk | Description |
|------|-------------|
| Misconfigured templates | One bad template deploys insecure config at scale across all environments |
| Secrets in code | API keys or passwords hardcoded in IaC files = credential exposure in version control |
| Uncontrolled repositories | IaC repos must be access-controlled; treat infrastructure code as sensitive |
| Drift | Manual changes to deployed infrastructure diverge from the template; security controls may be weakened |

- **Version control is required**: enables rollback, change auditing, and peer review of infrastructure changes

## Serverless / FaaS (Function as a Service)

- Code runs on-demand in short-lived execution environments; no server to provision, patch, or manage
- Provider manages all underlying infrastructure; customer only manages function code and triggers
- Examples: AWS Lambda, Azure Functions, Google Cloud Functions
- **Reduced attack surface**: no OS to patch, no persistent server to compromise or enumerate

### Serverless Security Risks

- **Function injection**: malicious input exploits function logic — similar to injection attacks in traditional apps
- Over-permissive IAM roles attached to functions enable privilege escalation if function is compromised
- Event-driven execution makes audit logging and forensic reconstruction harder
- Short execution windows and cold-start behavior complicate security monitoring

---

## Microservices Architecture

- Application built as a collection of small, loosely coupled, independently deployable services
- Services communicate via **APIs** (REST, gRPC) or message queues (Kafka, RabbitMQ)
- Each service handles one business function; can be scaled or updated independently
- Typically container-based (Docker + Kubernetes)

### Microservices Security Considerations

- **API security is critical**: every service interaction is an API call; each is an attack surface
- Service-to-service authentication is required (mutual TLS / mTLS via a service mesh like Istio)
- Harder to monitor — internal traffic between services requires distributed tracing or service mesh visibility
- Blast radius of a single compromise is smaller but lateral movement between services is a real risk

## Network Infrastructure Models

### Physical Isolation / Air Gap

- Network is completely disconnected from the internet and all external networks
- Highest level of security; used for classified government systems, nuclear plant controls, and critical SCADA
- Data transfer must be physical (USB, optical disk) — known as **sneakernet**
- Air gap bridging attacks exist (acoustic covert channels, RF, optical — e.g., AirHopper, MOSQUITO)

### Logical Segmentation

- Software-defined separation of network traffic on shared physical infrastructure
- **VLANs** (Virtual LANs): segment traffic at Layer 2; different broadcast domains on the same switches
- **Subnets**: Layer 3 separation; ACLs and firewall rules enforce access boundaries between segments
- Cheaper than physical separation; still provides meaningful isolation for most environments

### SDN (Software Defined Networking)

- Separates the **control plane** (where/how to route traffic) from the **data plane** (forwarding traffic)
- Centralized controller manages network policies dynamically via software APIs
- More flexible and programmable than traditional hardware-only networks
- **Security implication**: centralized controller is a high-value target — compromising it means controlling the entire network

## On-Premises vs Cloud

| Attribute | On-Premises | Cloud |
|-----------|------------|-------|
| Control | Full | Shared (varies by model) |
| Responsibility | Full | Shared |
| Cost model | Capital expense (CapEx) | Operational expense (OpEx) |
| Scalability | Limited by physical hardware | Elastic on demand |
| Patch management | Customer-owned entirely | Shared per service model |

---

## Centralized vs Decentralized Architecture

| Attribute | Centralized | Decentralized |
|-----------|------------|--------------|
| Management | Easier — single point of control | Harder — distributed management required |
| Monitoring | Simpler — all traffic flows through center | Complex — visibility spread across nodes |
| Resilience | Single point of failure risk | More resilient — no single failure point |
| Examples | Traditional corporate DC, central SIEM | Blockchain, distributed mesh networks |

## Containerization

- **Containers**: isolated process environments sharing the host OS kernel (unlike VMs which have full separate kernels)
- **Docker**: most common container runtime; packages app + dependencies into portable, reproducible images
- **Kubernetes (K8s)**: orchestrates containers at scale; manages deployment, scaling, and networking

### Container Security Risks

| Risk | Mitigation |
|------|-----------|
| Vulnerable base images from public registries | Scan images with Trivy or Clair; use trusted base images |
| Registry compromise | Restrict push access; sign images; use private registries |
| Container running as root | Run as non-root user; enforce pod security policies |
| Container breakout to host | Keep host OS and container runtime patched; use namespaces/cgroups |

---

## Virtualization

- **Hypervisor**: software layer creating and managing Virtual Machines (VMs)

| Type | Description | Examples |
|------|-------------|---------|
| **Type 1** (Bare metal) | Runs directly on hardware; no host OS required | VMware ESXi, Microsoft Hyper-V, Xen |
| **Type 2** (Hosted) | Runs on top of a host OS | VirtualBox, VMware Workstation |

- Type 1 is more secure — smaller attack surface with no underlying host OS to compromise
- **VM escape**: attacker breaks out of a VM to the hypervisor or another VM — critical, high-severity vulnerability
- **Snapshot**: point-in-time copy of a VM; can revert to clean state; useful for recovery and malware analysis sandboxing

---

## IoT (Internet of Things)

- Devices with limited compute: sensors, cameras, smart appliances, industrial monitors, wearables
- **Security weaknesses**:
  - Default credentials (admin/admin) rarely changed by users
  - Infrequent or impossible patching — firmware update mechanisms often absent
  - Protocol weaknesses: Zigbee, Z-Wave, MQTT often lack encryption or authentication
  - No traditional endpoint security (AV, EDR) available for most devices
  - Massive scale = massive attack surface expansion

---

## Industrial / OT / ICS / SCADA

| Term | Meaning |
|------|---------|
| **OT** (Operational Technology) | Hardware/software controlling physical processes in manufacturing, utilities |
| **ICS** (Industrial Control Systems) | Broad term for industrial automation and control |
| **SCADA** (Supervisory Control and Data Acquisition) | Monitors/controls large-scale distributed systems: power grids, water treatment, pipelines |
| **RTOS** (Real-Time Operating System) | OS with deterministic, time-bounded responses; used in medical devices, vehicles, robotics |
| **Embedded Systems** | Single-purpose devices with fixed firmware; printers, routers, vehicle ECUs |

- **Stuxnet (2010)**: nation-state malware targeting Iranian nuclear facility SCADA; destroyed centrifuges via Siemens PLCs
- **Safety-critical**: ICS/SCADA compromise can cause physical harm, injury, or death — not just data loss
- Legacy systems often cannot be patched without operational downtime — operational continuity prioritized over patching
- Air gap is the preferred control; when network-connected, use **data diodes** (unidirectional gateways)

---

## High Availability

| Concept | Description |
|---------|-------------|
| **Load Balancing** | Distributes traffic across multiple servers; removes single server as a point of failure |
| **Clustering** | Multiple systems share workload and state; one node fails, others continue without interruption |
| **Redundancy** | Duplicate components (dual power supplies, dual NICs, RAID storage) eliminate single points of failure |

## Exam Tips

- Know the cloud service model layers cold: IaaS = you manage OS up; PaaS = you manage app + data; SaaS = you manage data + access only
- Shared responsibility: the CSP is NEVER responsible for customer data classification, access policy, or encryption key management — those always belong to the customer
- IaC risk = misconfigured templates deploy at scale; hardcoded secrets in code = credential exposure; version control is the primary mitigation
- Air gap = highest security; used for SCADA/classified systems; physical media transfer is the only data path
- VM escape is the critical hypervisor risk; Type 1 hypervisors are more secure than Type 2 (no host OS layer)
- IoT = default credentials + no patching + weak protocols; SCADA/ICS = physical safety risk, not just data risk
- Stuxnet is the canonical ICS attack example — targeted Iranian nuclear centrifuges via Siemens S7 PLCs
- SDN centralizes control — the SDN controller becomes the highest-value attack target on the network
- Microservices = API surface is the primary attack surface; mTLS required for service-to-service auth
- Serverless reduces OS attack surface but introduces function injection risk and IAM over-permission as new threats
