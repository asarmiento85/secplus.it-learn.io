# 4.4 — Security Alerting and Monitoring

## What you need to know

Security monitoring is the continuous collection, correlation, and analysis of data from systems, applications, and infrastructure to detect threats and ensure compliance. The exam tests which tools are used for what purpose, how SIEM works, the difference between agent-based and agentless monitoring, and the workflow for responding to alerts. You should know the specific purpose of each monitoring tool and be able to choose the right one for a given scenario.

---

## What to Monitor

| Category | Examples |
|---|---|
| **Systems** | OS events, authentication, process execution, patch status |
| **Applications** | Errors, transactions, login failures, API calls |
| **Infrastructure** | Network devices, firewalls, switches, routers, cloud services |
| **User activity** | Login times, file access, privilege use, data transfers |
| **Data** | Sensitive data movement, DLP alerts, encryption status |

---

## Core Monitoring Activities

| Activity | Description |
|---|---|
| **Log aggregation** | Centralize logs from all sources into a single platform (SIEM) |
| **Alerting** | Automatic notification when a threshold or anomaly is detected |
| **Scanning** | Vulnerability scanning and compliance scanning |
| **Reporting** | Scheduled or on-demand reports for compliance and management |
| **Archiving** | Long-term log storage for forensics and compliance |
| **Alert response** | Investigate, triage, and remediate triggered alerts |

### Alert Response Workflow
1. **Validate** — confirm the alert is real (true positive) vs. false positive
2. **Triage** — assess severity and prioritize response based on impact
3. **Investigate** — gather additional context; correlate with other data
4. **Contain** — if confirmed threat, isolate affected systems
5. **Remediate** — eradicate threat, restore systems, apply fixes
6. **Document** — record findings; update playbooks if needed

---

## Alert Types

| Type | Description | Example |
|---|---|---|
| **Threshold-based** | Fires when a metric exceeds a defined limit | >10 failed logins in 1 minute |
| **Anomaly-based** | Fires when behavior deviates from established baseline | User logs in from new country at 3 AM |
| **Signature-based** | Matches known attack patterns | IDS signature matches known exploit |
| **Correlation-based** | Multiple events combined trigger alert | Failed login + new process + outbound connection |

## Key Tools and Standards

### SCAP (Security Content Automation Protocol)
- **NIST standard** for automating vulnerability checking and compliance assessment
- Provides machine-readable formats: **OVAL** (vulnerability definitions), **XCCDF** (checklists), **CVE**, **CVSS**
- Enables automated comparison of system configurations against benchmarks
- Used by tools like **OpenSCAP**, DISA STIGs

### CIS Benchmarks
- **Center for Internet Security** — community-developed configuration standards
- Provides hardening guides for OS, cloud, applications, network devices
- Two implementation levels: **Level 1** (basic, minimal impact) and **Level 2** (defense-in-depth, may impact functionality)

### Agent vs. Agentless Monitoring

| Approach | How It Works | Pros | Cons |
|---|---|---|---|
| **Agent-based** | Software installed on each endpoint; collects and forwards data | More detail; works offline; deeper OS visibility | Management overhead; not possible on all devices |
| **Agentless** | Uses existing protocols (WMI, SSH, SNMP) to pull data | No installation needed; works on network devices | Less detail; requires network access; polling delays |

## SIEM (Security Information and Event Management)

SIEM is the central nervous system of a security monitoring program.

| SIEM Capability | Description |
|---|---|
| **Log collection** | Ingest logs from all sources (firewalls, endpoints, apps, cloud) |
| **Normalization** | Convert logs from different formats into a common schema |
| **Correlation** | Match related events across sources to detect complex attacks |
| **Alerting** | Trigger alerts when correlation rules fire |
| **Dashboards** | Real-time visual overview of security posture |
| **Log storage / Archiving** | Long-term retention of events for forensics and compliance |
| **Search and query** | Analysts can search historical logs to investigate incidents |

- Popular SIEM platforms: **Splunk**, **Microsoft Sentinel**, **IBM QRadar**, **Elastic SIEM**
- **Log retention** in SIEM is critical for forensics — events older than retention window are gone
- SIEM ingests raw logs and requires **tuning** to reduce false positives

## Specific Monitoring Tools

### Antivirus / EDR
- **Antivirus**: signature-based detection of known malware
- **EDR** (Endpoint Detection and Response): behavioral detection, threat hunting, automated response
- EDR generates rich endpoint telemetry that feeds into SIEM

### DLP (Data Loss Prevention)
- Monitors for **sensitive data leaving the organization**
- Inspects: email attachments, web uploads, USB transfers, cloud sync
- Can **block** or **alert** on policy violations
- Modes: **Network DLP** (perimeter), **Endpoint DLP** (device), **Cloud DLP** (SaaS)

### SNMP Traps
- **Simple Network Management Protocol** — used to manage network devices
- **SNMPv3** = authenticated and encrypted (use this version; v1/v2 are insecure)
- **SNMP Traps** = unsolicited alerts sent by network devices to the SNMP manager when events occur
- Example: router sends trap when interface goes down or CPU exceeds 90%

### NetFlow
- **Network traffic metadata analysis** — records: source/dest IP, ports, protocol, byte/packet counts, timestamps
- Does **NOT capture packet contents** — only metadata
- Used to: detect anomalies, identify top talkers, investigate suspicious connections, bandwidth monitoring
- Similar technologies: **sFlow** (sampled), **IPFIX** (NetFlow v10 standard), **J-Flow** (Juniper)

### Vulnerability Scanners
- **Nessus** (Tenable) — industry-leading commercial vulnerability scanner
- **Qualys** — cloud-based vulnerability management platform
- **OpenVAS** — open-source vulnerability scanner (Greenbone)
- Output: list of CVEs found, severity, affected systems, remediation guidance

## Log Aggregation Architecture

```
Endpoints → Log Forwarders → SIEM (collect, normalize, correlate) → Alerts → SOC Analysts
Network Devices → Syslog → SIEM
Cloud Services → API → SIEM
```

- **Syslog** (port 514 UDP; 6514 TLS) — standard log transport protocol for network devices and Linux systems
- **Windows Event Forwarding (WEF)** or **Winlogbeat** — forward Windows events to SIEM
- **Log forwarders** (Fluentd, Logstash, Splunk Universal Forwarder) — collect and ship logs

## Exam Tips

- **SIEM** aggregates, normalizes, correlates, and alerts — it is the central hub of security monitoring.
- **NetFlow** captures metadata, NOT packet contents — if you need full packet data, use a **packet capture (PCAP)**.
- **SNMP Traps** are sent by the device (unsolicited) vs. SNMP polling (manager requests status).
- **SNMPv3 is the only secure version** — v1 and v2 transmit community strings in plaintext.
- **DLP** is specifically about preventing data exfiltration — it monitors for sensitive data leaving.
- **Agent-based** monitoring gives more detail but requires installation; **agentless** is easier to deploy but less thorough.
- **CIS Benchmarks** = configuration standards (Level 1 = basic, Level 2 = advanced).
- **SCAP** = the NIST standard for automated compliance checking — enables tool interoperability.
- **Alert tuning** is critical — too many false positives cause alert fatigue; too few miss real threats.
- **EDR** is behavioral/advanced; **antivirus** is signature-based — EDR is the evolution of AV.
- The alert workflow is: Validate → Triage → Investigate → Contain → Remediate → Document.
