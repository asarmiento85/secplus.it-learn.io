# 4.8 — Incident Response Activities

## What you need to know

Incident response is the structured approach to handling security incidents. Security+ tests your knowledge of the IR lifecycle phases (PICERL), the difference between training types, digital forensics concepts (especially chain of custody and evidence acquisition), root cause analysis, and threat hunting. Expect scenarios asking what an IR analyst should do at a specific phase, or which forensic procedure preserves evidence integrity.

---

## Incident Response Lifecycle (PICERL)

This is the most commonly tested framework. Memorize the phases and what happens in each.

| Phase | Key Activities |
|---|---|
| **P — Preparation** | IR plan, IR team (CSIRT), tools and jump bags, training, tabletop exercises, legal agreements |
| **I — Identification (Detection)** | Detect the incident via alerts, user reports, monitoring; determine if it's a real incident |
| **C — Containment** | Stop the spread; isolate affected systems; short-term (immediate) and long-term (stable) containment |
| **E — Eradication** | Remove the threat: malware removal, close vulnerabilities, patch systems, remove attacker footholds |
| **R — Recovery** | Restore systems to production; verify systems are clean; monitor for recurrence |
| **L — Lessons Learned** | Post-incident review; what happened, timeline, what worked, what failed, how to prevent recurrence |

### NIST SP 800-61 Phases (Alternative Framework)
1. Preparation
2. Detection and Analysis
3. Containment, Eradication, and Recovery
4. Post-Incident Activity (Lessons Learned)

---

## Preparation Phase

- **IR Plan** — documented procedures for handling security incidents
- **CSIRT** (Computer Security Incident Response Team) — designated team with defined roles
- **Incident severity classification** — P1 (critical) through P4 (informational)
- **Communication plan** — who to notify (legal, HR, executives, law enforcement, regulators, public)
- **Tools** — forensic workstations, write blockers, forensic software (FTK, Autopsy), network capture tools
- **Legal agreements** — retain outside counsel; establish law enforcement relationships in advance
- **Evidence handling procedures** — chain of custody forms, evidence bags, tamper-evident seals

---

## Detection and Identification

- **Sources**: SIEM alerts, EDR alerts, user reports, threat intelligence, external notifications
- **Triage**: Is this a true positive? What systems are affected? What data is at risk?
- **Incident declaration**: formally declare an incident and activate the IR plan
- **Scope determination**: identify all affected systems, users, and data
- **Severity assessment**: is this a minor incident or a major breach?

---

## Containment

### Short-Term Containment
- Immediate action to stop the spread **right now**
- Examples: isolate affected system from network, block attacker's IP, disable compromised account
- Goal: stop bleeding while preserving evidence

### Long-Term Containment
- More stable solution that can be maintained while eradication is prepared
- Examples: temporary network segmentation, monitoring compromised system in isolated VLAN
- May continue for days/weeks while thorough eradication is planned

**Key principle**: Containment must **balance stopping the attack with preserving evidence**.

---

## Eradication

- **Remove malware**: scan and clean all affected systems
- **Close initial access vector**: patch the exploited vulnerability, fix misconfiguration
- **Remove attacker footholds**: backdoors, scheduled tasks, rogue user accounts, persistence mechanisms
- **Rebuild if necessary**: if a system cannot be fully cleaned, reimage from known-good baseline
- **Verify clean**: use forensic tools to confirm threat is fully removed

---

## Recovery

- **Restore from clean backup** or reimage system
- **Verify integrity**: confirm systems are operating correctly before returning to production
- **Monitor closely**: watch for signs of reinfection or attacker return for 30-90 days
- **Return to production**: bring systems back in phases; high-criticality systems last
- **Credential resets**: force password resets for all potentially compromised accounts

---

## Lessons Learned / Post-Incident Review

- Held within 2 weeks of incident closure while memory is fresh
- Produces a **Post-Incident Report (PIR)**
- Topics covered: what happened, timeline, root cause, what was done well, what should improve, action items
- Action items fed back into: IR plan updates, tool purchases, training, policy changes
- **Non-punitive culture** — goal is learning, not blame

---

## Training and Testing

| Type | Description | Involves Real Systems? |
|---|---|---|
| **Security awareness training** | General education for all employees | No |
| **Role-based training** | Training specific to job function (e.g., analyst, developer, executive) | No |
| **Tabletop exercise** | Discussion-based scenario walkthrough; no systems involved | No |
| **Simulation** | Realistic scenario test; may involve actual tools and responses | Yes (controlled) |
| **Red team / Blue team** | Adversarial test: red attacks, blue defends | Yes |
| **Purple team** | Red and blue work together to improve both offense and defense | Yes |

- **Tabletop** = cheapest and safest; good for testing decision-making and communication
- **Simulation** = more realistic; tests actual technical procedures

## Root Cause Analysis (RCA)

- Goal: identify the **underlying cause** of an incident, not just the symptoms
- **5 Whys** method: ask "why?" repeatedly until root cause is found

  Example: Ransomware encrypted files → Why? Phishing email executed → Why? User clicked link → Why? No security awareness training → Why? Training program not funded → **Root cause: lack of investment in training**

- **Fishbone diagram** (Ishikawa) — visual cause-and-effect analysis; categories: People, Process, Technology, Environment

---

## Threat Hunting

- **Proactive** search for threats that have **not yet been detected** by automated tools
- Assumption: "we have already been compromised; where is the threat hiding?"
- Types:
  - **IOC-based**: search for known indicators (IP addresses, file hashes, domain names)
  - **TTP-based**: search for attacker techniques based on threat intelligence (MITRE ATT&CK)

### Threat Hunting Process
1. **Hypothesis**: form a theory about where a threat might be (e.g., "APT group uses PowerShell for lateral movement")
2. **Hunt**: search SIEM, EDR telemetry, logs for evidence
3. **Discover**: find or rule out the threat
4. **Improve**: if found, respond; always feed findings back into detection rules

## Digital Forensics

### Core Principles

| Principle | Description |
|---|---|
| **Preserve original evidence** | Never work on original media; always use a forensic copy |
| **Chain of custody** | Document every person who handled evidence, when, and why |
| **Write blockers** | Hardware or software device that prevents any writes to evidence media |
| **Forensic copy (image)** | Bit-for-bit copy of storage media; captures deleted files, slack space |

### Evidence Acquisition Process
1. **Legal hold** — suspend normal deletion/retention policies; preserve all potentially relevant data
2. **Identify** — determine what evidence exists and where
3. **Collect** — acquire forensic image using write-blocked tools (FTK Imager, dd command)
4. **Verify** — hash the original and the copy (MD5/SHA-256); hashes must match
5. **Store** — secure evidence with tamper-evident controls; document storage location
6. **Analyze** — work only on the forensic copy
7. **Report** — document findings in court-admissible format
8. **Present** — testify or provide findings to legal team

### Forensic Tools

| Tool | Description |
|---|---|
| **FTK Imager** | Acquire forensic images; free; widely used |
| **Autopsy** | Open-source forensic analysis platform |
| **dd** (Linux) | Command-line disk imaging: `dd if=/dev/sda of=/forensic/image.dd` |
| **Volatility** | Memory forensics analysis framework |
| **Wireshark** | Packet capture analysis |
| **EnCase** | Commercial forensic suite; court-accepted |

### Chain of Custody
- **Documents every transfer of evidence** — from acquisition to courtroom
- Includes: who collected it, date/time, condition, who transported it, storage location
- Any **break in chain of custody** can make evidence inadmissible in court
- Tamper-evident bags, seals, and secure storage are required

### Legal Hold
- **Suspends normal data retention/deletion** policies when litigation is anticipated
- All potentially relevant data must be preserved — even if normally scheduled for deletion
- Failure to preserve = **spoliation** — destruction of evidence; serious legal consequences
- IT receives legal hold notice from legal team; must implement immediately

### E-Discovery
- **Electronic discovery** — legal process of producing electronic evidence in litigation
- Tools: Relativity, Nuix, Logikcull
- Process: identify → preserve (legal hold) → collect → process → review → produce

## Exam Tips

- **PICERL order**: Preparation, Identification, Containment, Eradication, Recovery, Lessons Learned — memorize this.
- **Containment** happens BEFORE eradication — you stop the spread first, then remove the threat.
- **Chain of custody** — any break makes evidence inadmissible. Document every hand-off.
- **Legal hold** suspends deletion policies — triggered when litigation is anticipated, not after it starts.
- **Write blockers** are mandatory for evidence acquisition — prevent accidental modification.
- A **forensic copy** (bit-for-bit image) is used for analysis — the **original media is never touched**.
- **Hash verification** (MD5/SHA-256) proves the forensic copy is identical to the original.
- **Tabletop exercises** involve no actual systems — they test decision-making and communication.
- **Threat hunting is proactive** — it assumes compromise exists and searches for it, unlike reactive monitoring.
- **5 Whys** gets to root cause — keep asking "why" until the fundamental issue is identified.
- **Eradication** must be complete before recovery — returning a still-compromised system to production is worse.
- **Lessons learned** is a required phase — without it, the same incidents recur.
