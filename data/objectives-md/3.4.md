# 3.4 — Resilience and Recovery in Security Architecture

## What you need to know

Resilience and recovery architecture ensures that systems can survive failures, disasters, and attacks while minimizing downtime and data loss. The Security+ exam tests your knowledge of site types, backup strategies, high availability concepts, power resilience, and the key metrics (RTO, RPO, MTTR, MTBF) that define acceptable recovery parameters. Expect scenario questions asking you to select the right site type, backup strategy, or metric for a given business requirement.

---

## High Availability Concepts

High availability (HA) means designing systems to eliminate single points of failure (SPOFs) so that services remain available even when individual components fail.

### Load Balancing

| Mode | Description | Behavior on Node Failure |
|------|-------------|------------------------|
| **Active-Active** | All nodes serve live traffic simultaneously | Remaining nodes absorb the failed node's load |
| **Active-Passive** | One active node; one or more standby nodes | Standby promotes to active when primary fails |

- Active-Active provides better performance (all capacity utilized) and faster failover
- Active-Passive uses standby capacity that sits idle — simpler failover, higher cost per transaction capacity

### Clustering

- Multiple servers share a workload and maintain shared state; presented as a single logical system
- **Automatic failover**: if one cluster node fails, another immediately takes over without manual intervention
- Common in: database clusters (SQL Always On, Oracle RAC), application server clusters, virtualization (vSphere HA)

## Recovery Site Types

| Site Type | Configuration | Time to Operational | Cost | Data Currency |
|-----------|--------------|-------------------|------|--------------|
| **Hot site** | Fully operational duplicate; live data replication; systems powered on | Minutes to hours | Highest | Near real-time |
| **Warm site** | Partially configured; hardware present; some data; systems may be off | Hours to days | Medium | Hours to days old |
| **Cold site** | Empty facility with power and connectivity; no hardware or data | Days to weeks | Lowest | Requires restore from backup |

- **Hot site**: most expensive but fastest recovery; required when RTO is measured in minutes
- **Cold site**: cheapest but slowest; acceptable only when extended downtime is tolerable
- **Geographic dispersion**: recovery sites must be geographically separated from the primary site to avoid both being hit by the same regional disaster (hurricane, earthquake, power grid failure)

## Platform Diversity and Multi-Cloud

### Platform Diversity

- Using different operating systems, hardware vendors, or software versions across infrastructure
- Prevents a single vulnerability from compromising all systems simultaneously
- Example: if all servers run the same OS version, one zero-day vulnerability affects 100% of servers; diverse platforms reduce this risk
- Trade-off: increased management complexity

### Multi-Cloud Strategy

- Deploying workloads across multiple cloud providers (AWS + Azure + GCP)
- Benefits:
  - Avoids vendor lock-in — not dependent on a single provider's availability
  - Improved resilience — one provider outage does not take down all workloads
  - Can leverage best-of-breed services from each provider
- Challenges: consistent security policies, identity federation, cost management, and monitoring complexity

## Continuity of Operations (COOP)

- A formal plan ensuring that critical business functions continue during and after a disruptive event
- Identifies essential functions, personnel, resources, and alternate facilities
- Aligns with **BCP (Business Continuity Plan)**: the broader plan for maintaining operations
- **DRP (Disaster Recovery Plan)**: the technical subset focused on restoring IT systems
- COOP is the government-sector term; BCP is the corporate-sector term — they overlap significantly

---

## Capacity Planning

Organizations must plan for adequate capacity across three dimensions:

| Dimension | Consideration |
|-----------|--------------|
| **People** | Sufficient trained staff; cross-training; succession planning; vendor support contracts |
| **Technology** | Compute capacity (CPU, RAM); storage growth; database scaling; bandwidth |
| **Infrastructure** | Power capacity (UPS, generators, PDUs); cooling; physical rack space; network bandwidth |

- Capacity planning must account for peak loads during disaster recovery (when systems are running backup operations simultaneously with normal workloads)

## Resilience Testing Methods

| Method | Description | Systems Affected |
|--------|-------------|----------------|
| **Tabletop exercise** | Discussion-based walkthrough of a scenario; stakeholders talk through their responses | No systems involved |
| **Failover test** | Actually trigger failover to the secondary system to verify it works end-to-end | Production or test environment |
| **Simulation** | Realistic simulation of a disaster scenario; may involve actual system actions | Controlled test environment |
| **Parallel processing** | Run both primary and backup systems simultaneously; compare outputs for accuracy | Both primary and DR systems |

- Tabletop exercises are the safest and most common first step
- Failover tests are the most valuable — they verify the recovery actually works (documentation-only plans often fail in real disasters)
- Recovery plans must be **tested regularly** — a plan that has never been tested should not be trusted

## Backup Types and Strategies

### Backup Types

| Type | What Is Backed Up | Backup Speed | Restore Speed | Restore Requires |
|------|------------------|-------------|--------------|-----------------|
| **Full** | Everything | Slowest | Fastest | Just the full backup |
| **Incremental** | Changes since last backup (full or incremental) | Fastest | Slowest | Full + every incremental since |
| **Differential** | Changes since last full backup | Medium | Medium | Full + latest differential only |

- **Incremental** is space-efficient but restoration requires chaining many backup sets
- **Differential** restores faster than incremental (only two sets needed) but grows in size each day until the next full backup
- Most organizations use a combination: full weekly + incremental daily, or full + differential daily

### Backup Best Practices

| Practice | Description |
|----------|-------------|
| **3-2-1 Rule** | 3 copies of data, 2 different media types, 1 offsite |
| **Backup encryption** | All backup media must be encrypted — stolen tape = data breach |
| **Offsite storage** | At least one copy stored at a geographically separate location |
| **Immutable backups** | Backups that cannot be modified or deleted; protects against ransomware |
| **Test restores** | Periodically restore from backup and verify data integrity |
| **Frequency** | Determined by RPO — if RPO = 4 hours, backups must run every 4 hours |

### Snapshots

- Point-in-time copy of a volume, VM, or database state
- Created quickly (metadata-based on copy-on-write); minimal initial storage overhead
- Common in: VMware, AWS EBS, Azure Managed Disks, database systems
- Not a replacement for backups — snapshots live on the same storage system; a storage failure loses both

### Replication

| Type | Description | Use Case |
|------|-------------|---------|
| **Synchronous** | Write confirmed only after data written to both primary and replica | Zero RPO; used for financial systems; adds write latency |
| **Asynchronous** | Write confirmed at primary; replica updated shortly after | Near-zero RPO; lower latency impact; small data loss window |

- Replication provides geographic redundancy and rapid failover
- Does not protect against logical corruption (if you delete a record, the deletion replicates too) — use point-in-time snapshots or backups for that

### Journaling

- Maintains a log (journal) of all changes made to a file system or database
- Enables replay of changes to recover to a specific point in time after a failure
- File system journaling (ext4, NTFS): ensures filesystem consistency after a crash
- Database transaction logs: allow roll-forward or roll-back recovery

## Power Resilience

| Component | Function | Duration |
|-----------|----------|---------|
| **UPS** (Uninterruptible Power Supply) | Battery-backed power; provides clean power and brief runtime during outages | Minutes (10–30 typical); enough for graceful shutdown or generator start |
| **Generator** | Fuel-based power generation for extended outages | Hours to days (depends on fuel supply and refueling plan) |
| **Dual power supply** | Two independent power feeds to a device (from separate circuits/PDUs) | Continuous — one feed can fail without affecting the device |
| **PDU** (Power Distribution Unit) | Smart power strip for racks; provides remote monitoring, metering, and outlet control | Continuous distribution |

- **UPS + Generator**: the standard combination for data centers; UPS covers the gap while the generator starts (typically 10–30 seconds)
- **Dual supply**: servers, switches, and storage arrays should have redundant power supplies fed from different PDUs on different UPS/generator circuits

## Key Recovery Metrics

These four metrics are heavily tested — know each definition, the relationship between them, and which drives which business decisions.

| Metric | Full Name | Definition | Who Sets It |
|--------|-----------|------------|------------|
| **RTO** | Recovery Time Objective | Maximum acceptable downtime after a disaster before business impact becomes unacceptable | Business owners |
| **RPO** | Recovery Point Objective | Maximum acceptable data loss measured in time (how old can the restored data be?) | Business owners |
| **MTTR** | Mean Time To Repair | Average time to restore a failed component or system to full operation | IT/operations (measured) |
| **MTBF** | Mean Time Between Failures | Average time between failures; measure of system reliability | IT/operations (measured) |

### How They Relate to Architecture Decisions

| Requirement | Architecture Implication |
|-------------|------------------------|
| RTO = minutes | Hot site or active-active HA required |
| RTO = hours | Warm site acceptable |
| RTO = days | Cold site may be sufficient |
| RPO = zero | Synchronous replication required |
| RPO = 1 hour | Backups or async replication every hour |
| Low MTTR needed | Spare parts on hand; runbooks; trained staff |
| High MTBF needed | High-quality hardware; redundancy; proactive maintenance |

- **RTO drives the site type choice** (hot/warm/cold)
- **RPO drives the backup frequency and replication strategy**
- Lower RTO and RPO = higher cost — the business must decide the acceptable level of risk vs investment

## Exam Tips

- Hot site = immediate failover, most expensive; cold site = days to weeks, cheapest — match the site type to the stated RTO
- RTO = how long you can be down; RPO = how much data you can afford to lose — know both definitions cold
- Full backup = slowest to create, fastest to restore; incremental = fastest to create, slowest to restore (need full + all incrementals)
- The 3-2-1 backup rule: 3 copies, 2 media types, 1 offsite — this will appear on the exam
- Backup encryption is mandatory — an unencrypted stolen backup is a reportable data breach
- Synchronous replication = zero RPO but adds write latency; asynchronous = near-zero RPO with less latency
- UPS covers the gap during generator startup; together they provide continuous power resilience
- MTBF = how often does it break (reliability); MTTR = how long to fix it (recovery speed)
- Tabletop exercise = discussion only, no systems affected; failover test = actually triggers recovery systems
- Snapshots are NOT backups — they live on the same storage system and do not protect against storage failures
