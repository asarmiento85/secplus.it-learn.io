# 1.4 — Cryptographic Solutions

## What you need to know

Cryptography is one of the most concept-dense objectives in Domain 1. You need to understand how encryption protects data, how keys work, the difference between symmetric and asymmetric cryptography, how hashing provides integrity (not confidentiality), how certificates work, and how hardware and obfuscation techniques protect sensitive data. Expect at least 8-12 questions touching this objective across the exam.

---

## Public Key Infrastructure (PKI)

PKI is the framework of hardware, software, policies, and procedures needed to create, manage, distribute, and revoke digital certificates and key pairs.

| Key | Description | Used For |
|---|---|---|
| **Public Key** | Shared openly with anyone | Encrypting data TO the owner; verifying a digital signature |
| **Private Key** | Kept secret by the owner | Decrypting data; creating a digital signature |

**Memory rule:** Public key = anyone can use it; Private key = only the owner uses it.

### Key Escrow
- A backup copy of a private key is held by a **trusted third party**
- Purpose: enables recovery of encrypted data if the key owner loses their key
- Risk: the escrow holder becomes a high-value target; if compromised, all escrowed keys are exposed
- Used by: law enforcement (lawful access), enterprise key management

### Certificate Authority (CA) Hierarchy
- **Root CA** — top of the hierarchy; self-signed; often kept offline for security
- **Intermediate CA** — issued by root CA; signs end-entity certificates; online and operational
- **End-entity certificate** — issued to users, devices, or servers; what we see in browsers

## Encryption Types

### Symmetric Encryption
- **Same key** is used to both encrypt and decrypt
- **Fast** — suitable for bulk data encryption
- **Key distribution problem** — how do you securely share the key?

| Algorithm | Key Sizes | Notes |
|---|---|---|
| **AES** (Advanced Encryption Standard) | 128, 192, 256-bit | Current gold standard; used in WPA2, TLS, BitLocker |
| **3DES** (Triple DES) | 112/168-bit | Legacy; deprecated; uses DES three times |
| **ChaCha20** | 256-bit | Stream cipher; used in TLS 1.3; good on mobile |

### Asymmetric Encryption
- **Key pair** — public key and private key (mathematically linked)
- **Slower** — used for key exchange and digital signatures, not bulk data
- Solves the key distribution problem — share the public key freely

| Algorithm | Key Sizes | Notes |
|---|---|---|
| **RSA** | 2048+ bit | Most common; key exchange, signatures, certificates |
| **ECC** (Elliptic Curve Cryptography) | 256+ bit | Shorter keys, same strength as RSA 2048; preferred for mobile/IoT |
| **DSA** (Digital Signature Algorithm) | 2048+ bit | Signatures only; not encryption |

### Hybrid Encryption
- Uses **asymmetric** to securely exchange a **symmetric** key, then uses **symmetric** for bulk data
- This is how **TLS (HTTPS)** works:
  1. Client and server use asymmetric (RSA/ECDH) to establish a shared symmetric key
  2. All subsequent data is encrypted with symmetric (AES)
- Best of both worlds: security of asymmetric, speed of symmetric

## Key Exchange Algorithms

| Algorithm | Description |
|---|---|
| **Diffie-Hellman (DH)** | Allows two parties to establish a shared secret over an insecure channel without transmitting the secret |
| **ECDH (Elliptic Curve Diffie-Hellman)** | ECC version of DH; more efficient; used in modern TLS |
| **DHE / ECDHE** | Ephemeral versions — new key pair per session; provides **Perfect Forward Secrecy (PFS)** |

**Perfect Forward Secrecy (PFS):** Even if the server's long-term private key is compromised, past sessions cannot be decrypted because each session used a unique ephemeral key.

## Encryption Levels

Data can be encrypted at different granularities depending on the protection needed.

| Level | Description | Examples |
|---|---|---|
| **Full-Disk** | Entire drive encrypted; requires key to boot | BitLocker (Windows), FileVault (macOS) |
| **Partition** | A specific partition/volume encrypted | Selected volume on a drive |
| **File** | Individual files encrypted | EFS (Encrypting File System), PGP-encrypted files |
| **Volume** | A logical volume encrypted | VeraCrypt container |
| **Database** | Entire database encrypted | Transparent Database Encryption (TDE) |
| **Record / Field-level** | Individual fields in a database encrypted | Encrypting just the SSN column; credit card numbers |

## Hashing

**Hashing** is a one-way function that produces a **fixed-length digest** (hash value) from input data. It is used for **integrity**, not confidentiality — you cannot reverse a hash to get the original data.

| Algorithm | Output Size | Status |
|---|---|---|
| **MD5** | 128-bit | Broken / collision vulnerable — do not use for security |
| **SHA-1** | 160-bit | Deprecated — vulnerable to collision attacks |
| **SHA-256** (SHA-2 family) | 256-bit | Current standard; widely used |
| **SHA-3** | Variable (224-512-bit) | Newer; different internal design from SHA-2 |
| **NTLM** | 128-bit | Windows password hashing; legacy; vulnerable |
| **bcrypt** | 184-bit | Password hashing with built-in salting |

### Salting
- A **random value** (salt) is added to the password *before* hashing
- Each user gets a unique salt — identical passwords produce different hashes
- **Prevents rainbow table attacks** — pre-computed tables of hash values are useless when each hash is unique
- Salt is stored alongside the hash in the database

### Key Stretching
Intentionally slows down the hashing process to make brute-force attacks harder.

| Algorithm | Notes |
|---|---|
| **PBKDF2** | Password-Based Key Derivation Function 2; runs hash many times (iterations) |
| **bcrypt** | Adaptive; cost factor increases iterations as hardware gets faster |
| **Argon2** | Winner of the Password Hashing Competition; memory-hard (resists GPU attacks) |

## Digital Signatures

Digital signatures provide **integrity**, **authenticity**, and **non-repudiation**.

**How it works:**
1. Sender creates a hash of the message
2. Sender **encrypts the hash with their PRIVATE key** (this is the signature)
3. Sender sends the message + signature
4. Receiver decrypts the signature with the sender's **PUBLIC key** to get the hash
5. Receiver hashes the message independently and compares — if they match, integrity is confirmed and authenticity is proven

**What digital signatures prove:**
- **Integrity** — message was not altered in transit
- **Authenticity** — the message came from the claimed sender
- **Non-repudiation** — sender cannot deny signing it

## Hardware Security

| Device | Description | Use Case |
|---|---|---|
| **TPM** (Trusted Platform Module) | Chip on the motherboard; stores cryptographic keys; enables measured boot | BitLocker key storage; platform integrity; secure boot |
| **HSM** (Hardware Security Module) | Dedicated external device for cryptographic operations | CA key storage; payment processing; high-security key management |
| **Key Management System (KMS)** | Software platform for managing key lifecycle | Enterprise key management; cloud key management (AWS KMS, Azure Key Vault) |
| **Secure Enclave** | Isolated processor area within a main chip | iPhone Secure Enclave for biometrics/Face ID; protects keys even from the OS |

**TPM vs. HSM:**
- **TPM** is a chip on the motherboard, tied to one computer; relatively low cost
- **HSM** is a dedicated, external, high-assurance device; FIPS 140-2/3 certified; used where highest security is needed

## Obfuscation Techniques

| Technique | Description | Example |
|---|---|---|
| **Steganography** | Hiding data *inside* another file (image, audio, video) | Embedding a secret message in a JPEG image |
| **Tokenization** | Replacing sensitive data with a non-sensitive substitute token | Credit card number replaced with a random token in transaction systems |
| **Data Masking** | Obscuring sensitive data; partial visibility | Showing only last 4 digits of SSN; asterisks in password fields |

**Key distinction:**
- **Encryption** — data is transformed and can be reversed with the key
- **Tokenization** — data is replaced; original stored securely elsewhere; token is meaningless
- **Steganography** — data is hidden (not encrypted by itself); the existence of the message is concealed

## Blockchain

- **Distributed ledger** — copies maintained by all participants; no central authority
- **Immutable** — once a block is added, it cannot be changed without invalidating the chain
- **Each block contains:** data, timestamp, hash of the block, hash of the *previous* block (chain linkage)
- **Consensus mechanisms** — proof of work (PoW), proof of stake (PoS) — ensure agreement on valid state
- **Security property:** Tampering with any block changes its hash, which breaks the chain — detectable
- **Use cases:** cryptocurrency, supply chain tracking, digital contracts (smart contracts), certificate transparency

## Digital Certificates

Certificates bind a public key to an identity, vouched for by a Certificate Authority (CA).

### X.509 Standard
- Format used by all modern certificates
- Contains: public key, owner identity (Subject), CA identity (Issuer), validity period, serial number, CA digital signature

### Certificate Types by Validation Level
| Type | Validation Level | Indicator |
|---|---|---|
| **DV (Domain Validated)** | Confirms domain ownership only | Padlock in browser |
| **OV (Organization Validated)** | Confirms domain + organization identity | Padlock; org name viewable |
| **EV (Extended Validation)** | Rigorous identity verification | Padlock; formerly green bar |

### Certificate Revocation
| Method | Description |
|---|---|
| **CRL** (Certificate Revocation List) | Periodic list of revoked serial numbers; downloaded by clients; can be outdated |
| **OCSP** (Online Certificate Status Protocol) | Real-time status check for a single certificate; more current than CRL |
| **OCSP Stapling** | Server attaches a cached OCSP response to the TLS handshake; faster and more private |

### Special Certificate Types
| Type | Description |
|---|---|
| **Wildcard** | `*.domain.com` — covers all subdomains of one domain |
| **SAN** (Subject Alternative Name) | One cert covers multiple distinct domains |
| **Self-signed** | No CA vouches for it; signed by its own private key; not trusted by browsers by default |
| **CSR** (Certificate Signing Request) | Request submitted to a CA to obtain a certificate; contains public key and identity info |

## Exam Tips

- **Symmetric** = same key, fast, bulk encryption (AES). **Asymmetric** = key pair, slow, key exchange/signatures (RSA, ECC).
- **Hybrid encryption** = asymmetric to exchange the symmetric key, then symmetric for the data. This is how TLS works.
- **Hashing** = integrity only; one-way; cannot decrypt. Do not confuse with encryption.
- **Digital signature** = hash encrypted with the sender's **PRIVATE** key. Verified with the sender's **PUBLIC** key.
- **Public key encrypts** data TO someone; **private key decrypts** data FROM someone. Remember: public = for others to use; private = only you use.
- **TPM** is on the motherboard. **HSM** is an external dedicated device. HSM is higher assurance.
- **Salt** prevents rainbow table attacks by making identical passwords hash differently.
- **Key stretching** (bcrypt, Argon2, PBKDF2) makes brute force slower — the hash takes longer to compute.
- **Tokenization** is used in payment systems — the real card number is never stored by the merchant.
- **Steganography** hides the existence of the message; it does not inherently encrypt.
- **OCSP** is real-time certificate status; **CRL** is a periodic list — OCSP is more current.
- **Wildcard cert** covers `*.domain.com` but NOT `domain.com` itself or sub-subdomains like `a.b.domain.com`.
- **Key escrow** = backup copy of private key held by a third party — used for lawful access or enterprise recovery.
- **ECC** provides the same security as RSA with shorter keys — preferred for mobile and IoT.
- **Perfect Forward Secrecy** requires ephemeral key exchange (ECDHE/DHE) — past sessions safe even if private key is later compromised.
